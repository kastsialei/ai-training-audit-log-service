## Findings

1. **T-4 Scope contradicts design.md §3 on Flyway index creation mode** — section B, severity **FAIL**.
   - **Where:** `tasks.md → T-4 "Scope"` and `tasks.md → T-4 "Notes / risks"`.
   - **What's wrong:** T-4 instructs the implementer to *"Use `-- flyway:executeInTransaction=false` and `CREATE INDEX CONCURRENTLY`"* and amplifies it in Notes/risks (*"Flyway transaction opt-out is the important gotcha. Never use a plain transactional `CREATE INDEX` here."*). `design.md → §3 "Estimated runtime"` says the inverse: *"indexes are created with plain `CREATE INDEX` in a transactional migration … `CONCURRENTLY` would only add the well-known HikariCP-`WaitForOlderSnapshots` hang risk in Spring Boot startup without buying anything for an empty table."*
   - **Why it matters:** REVIEW.md → B requires tasks to be consistent with design; in this project, the contradiction has a documented operational cost — the HikariCP snapshot hang during `mvn verify` — that the design was specifically updated to avoid. A future implementer following T-4 verbatim would re-introduce the hang.
   - **Suggested fix:** Rewrite T-4 Scope to *"Add `V3..V6__create_audit_events_query_indexes.sql` using plain `CREATE INDEX` in transactional migrations (no `CONCURRENTLY`, no `executeInTransaction=false`) per `design.md → §3 'Estimated runtime'`."* Replace the Notes/risks paragraph with the inverse caution: *"Plain `CREATE INDEX` is correct here because the table is empty when V3–V6 run. Do not switch to `CONCURRENTLY` for these — see `design.md § 3`. A later index added to the populated production table belongs in a separate migration that does use `CONCURRENTLY`."*

2. **V3-vs-V3–V6 migration version inconsistency** — section B, severity **WEAK**.
   - **Where:** `design.md → §3 "New indexes (V3 migration)"` vs. `design.md → §10 "Rollout"` and `tasks.md → T-4 Scope`.
   - **What's wrong:** §3 heading and migration filename example treat the work as a single migration `V3__create_audit_events_query_indexes.sql`, while §10 ("V3–V6 migrations") and the actual ship pattern split it into four migrations (one per index, V3..V6). T-4 reuses the single-file phrasing.
   - **Why it matters:** A reviewer cross-referencing §3 and §10 cannot tell whether the contract is one migration or four. Both layouts are defensible; the spec needs to commit to one so the migration filenames are predictable from the spec alone.
   - **Suggested fix:** Pick the four-file layout (already implied by §10 and matching the implementation) and update §3 to list the four expected filenames `V3__…`, `V4__…`, `V5__…`, `V6__…`, then mirror that in T-4 Scope.
