# T-7: Implement Query Use Case and Reader Port

**Refs:** US-1, US-3, US-4 ([requirements.md](../requirements.md)), [design.md](../design.md) section 4 "Where rules live", section 5 "Invariants", section 6 "Cursor format", section 2 "Public surface" (pagination semantics)
**Glossary:** [../glossary.md](../glossary.md) — canonical names, types, exceptions, problem types
**Depends on:** T-5 (criteria records + no-filter policy), T-6 (`Cursor` + `CursorCodec` + fingerprint)
**Est. LOC:** ~330

## Goal

Add the application-layer orchestrator that turns a validated query into a
page of audit events plus an opaque `next_cursor`. After this task, given
a `QueryCriteria` (from T-5), a `RequestedLimit`, and an optional incoming
cursor, the use case:

1. enforces the `≥1 substantive filter` policy from T-5 (design §4),
2. decodes the cursor through T-6's `CursorCodec` in a single call that
   both decodes and verifies the fingerprint matches the *current* filter
   set (design §5 invariant 3),
3. asks the persistence-agnostic `EventReader` port for `limit + 1` rows
   starting after the keyset position,
4. preserves the reader's row order (design §5 invariant 1 — the reader
   is the sort SoT, the use case must not re-sort),
5. encodes a fresh `next_cursor` from the **last returned page item**
   only when another page exists, otherwise returns `null`/empty.

No Spring Web, no JPA, no controller. The port is the seam T-8 plugs into.

## Approach

- New package `application/query/` (created here if T-5 has not already
  laid it down). Layering rule: this layer may depend on `domain/shared/`
  (`AuditEvent`, `Outcome` — promoted in T-3) and on T-5/T-6 types, but
  **must not** see HTTP, JPA, or Spring Web. `@Service` is allowed
  (Spring core only), mirroring `IngestEventUseCase`.
- Use case class `QueryAuditEventsUseCase`, constructor-injected with the
  three collaborators owned by sibling tasks:
  - `EventReader reader` — new port, defined in this task.
  - `CursorCodec codec` — from T-6 (provides `decode(raw, expectedFingerprint)`,
    `encode(Cursor)`, and `fingerprintFor(QueryCriteria)`).
  - `QueryCriteriaValidator validator` — from T-5.
- New port `EventReader` (interface) in `application/query/`. Exact
  signature per glossary §3:
  ```java
  List<AuditEvent> read(QueryCriteria criteria,
                        Optional<KeysetPosition> position,
                        int rowLimit);
  ```
  - `criteria` — the validated filter set from T-5.
  - `position` — `Optional<KeysetPosition>` (not nullable).
    `Optional.empty()` means "first page, no keyset predicate";
    `Optional.of(pos)` means "rows strictly before `pos`".
  - `rowLimit` — already includes the `+1` overshoot. The reader does
    **not** know about pagination semantics; it just returns up to N
    rows in canonical sort order. Keeping the port narrow is the T-7
    Notes constraint from `tasks.md`.
- New value record (glossary §2):
  `record KeysetPosition(OffsetDateTime recordedAt, UUID id)`
  in `application/query/`. Carries the strict-less-than anchor for the
  predicate T-8 will compile to SQL. Built from the decoded `Cursor`'s
  `rt`/`id` fields.
- New value record (glossary §2):
  `record QueryPage(List<AuditEvent> items, Optional<String> nextCursor)`
  in `application/query/`. The use case's public return type. Items are
  in reader order; `nextCursor` is `Optional.empty()` on the last page
  (also when items are empty).
- Public method signature per glossary §3 — `limit` is a separate
  parameter and is never read off `criteria`:
  ```java
  public QueryPage execute(QueryCriteria criteria,
                           RequestedLimit limit,
                           Optional<String> rawCursor);
  ```
- **Canonical execution order (glossary §7, locked):**
  1. `QueryCriteriaValidator.validate(criteria)` — throws
     `InvalidQueryException` with `ProblemKind.NO_FILTER`,
     `ProblemKind.BLANK_FILTER`, or `ProblemKind.INVALID_TIME_RANGE`.
  2. If `rawCursor.isPresent()`:
     ```java
     String expected = codec.fingerprintFor(criteria);
     Cursor decoded = codec.decode(rawCursor.get(), expected);
     ```
     This is a **single call**: `decode(raw, expectedFingerprint)`
     both base64-decodes and verifies the fingerprint, throwing
     `InvalidCursorException` with `ProblemKind.INVALID_CURSOR` for
     malformed input or `ProblemKind.CURSOR_FILTER_MISMATCH` when
     `decoded.fp` ≠ `expected`. There is no separate
     `verifyFingerprint` step.
  3. Build `Optional<KeysetPosition>` from the decoded cursor:
     `Optional.of(new KeysetPosition(decoded.rt(), decoded.id()))`
     when a cursor was supplied, otherwise `Optional.empty()`.
  4. `List<AuditEvent> rows = reader.read(criteria, position, limit.value() + 1);`
  5. Split rows: `items = rows.subList(0, min(rows.size(), limit.value()))`
     and `hasNext = rows.size() > limit.value()`.
  6. If `hasNext`: encode `next_cursor` from the **last item of
     `items`** (not from the dropped overshoot row), reusing the
     `expected` fingerprint when step 2 ran, or computing
     `codec.fingerprintFor(criteria)` fresh when step 2 was skipped.
  7. Return `new QueryPage(items, nextCursor)`.

  The reader is **not** called when steps 1 or 2 throw — verified by
  DoD tests `rejectsCriteria_whenNoSubstantiveFilterIsPresent` and
  `rejectsCursor_whenFingerprintMismatch_andDoesNotCallReader`, both
  of which assert `fake.callCount == 0`.
- "Reader order" preserved: the use case never sorts. The list returned
  to callers is a sublist of the reader's response in original order.
  This locks design §5 invariant 1 (sort stability) onto the reader's
  SQL `ORDER BY`, where it belongs.

## Step-by-step execution

Test-first per Ground rule 6. The fake reader is hand-rolled (not
Mockito) per AGENTS.md "simplicity first" — it records the most recent
call's arguments and exposes a `callCount` field used in the
fingerprint-mismatch test.

1. **Scaffold the fake.** Inside `QueryAuditEventsUseCaseTest`, add a
   nested `FakeEventReader implements EventReader` that:
   - holds a `List<AuditEvent>` to return,
   - captures the last `criteria`, `position`, `rowLimit`,
   - increments a `callCount` on every `read(...)`.
   No production code yet — this only compiles after step 2.

2. **Failing test (DoD #1, criteria mapping):**
   `passesCriteriaAndKeysetThroughToReader_whenCursorMatches`. Build a
   `QueryCriteria` with an `actor` and a `from`/`to`, encode a cursor
   from the same criteria via the real `CursorCodec`, call
   `useCase.execute(criteria, new RequestedLimit(50), Optional.of(rawCursor))`.
   Assert:
   - `fake.lastCriteria` is the same instance passed in.
   - `fake.lastPosition` is `Optional.of(new KeysetPosition(decoded.rt(), decoded.id()))`.
   - `fake.lastRowLimit == limit.value() + 1`.
   - `fake.callCount == 1`.
   Run — fails (no `QueryAuditEventsUseCase` yet).

3. **Implement the minimal use case.** Create `EventReader`,
   `KeysetPosition`, `QueryPage`, `QueryAuditEventsUseCase` with the
   validate → fingerprint+decode → position → reader call → pagination
   split logic per the canonical execution order. Keep the constructor
   `(EventReader, CursorCodec, QueryCriteriaValidator)`. The public
   method is
   `QueryPage execute(QueryCriteria criteria, RequestedLimit limit, Optional<String> rawCursor)`
   — `limit` is taken from the parameter, never from `criteria`.
   Re-run — green.

4. **Failing test (DoD #1, `limit + 1` pagination — more pages):**
   `dropsOvershootRow_andEncodesNextCursorFromLastReturnedItem`. Seed
   the fake with `limit + 1` events (e.g. `limit=2`, 3 events) in
   canonical order (newest first). Call
   `execute(criteria, new RequestedLimit(2), Optional.empty())`. Assert:
   - `result.items().size() == 2`.
   - `result.items()` equals the first two seeded events in order
     (no re-sort).
   - `result.nextCursor().isPresent()`.
   - Decode the returned cursor with the real `CursorCodec` (using
     `codec.fingerprintFor(criteria)` as the expected fingerprint);
     assert `decoded.rt()` and `decoded.id()` equal **the second**
     seeded event (the last item of the returned page, not the
     dropped third).
   - `decoded.fp()` equals `codec.fingerprintFor(criteria)`.
   Run — should pass if step 3 implemented the split correctly; if
   not, fix the split.

5. **Failing test (DoD #2, last page):**
   `returnsNullNextCursor_whenReaderReturnsAtMostLimitRows`. Two sub-cases
   in one test (parameterized or two `@Test` methods, whichever is
   simpler):
   - Reader returns exactly `limit.value()` rows → `nextCursor` is empty.
   - Reader returns fewer than `limit.value()` rows (incl. 0) →
     `nextCursor` is empty; `items` matches the reader output in order.
   Run — green when the split treats `rows.size() <= limit.value()` as
   "last page".

6. **Failing test (DoD #3, fingerprint mismatch short-circuits the
   reader):** `rejectsCursor_whenFingerprintMismatch_andDoesNotCallReader`.
   Build criteria A, encode a cursor from criteria A using the real
   codec, then build criteria B that differs in one substantive filter
   (e.g. different `actor`). Call
   `execute(criteriaB, new RequestedLimit(50), Optional.of(rawA))`.
   Assert:
   - throws `InvalidCursorException` with
     `ProblemKind.CURSOR_FILTER_MISMATCH` (the single-call
     `decode(raw, expectedFingerprint)` raises this when fingerprints
     disagree).
   - `fake.callCount == 0` (the reader was never invoked).
   Run — green if step 3 ordered the decode call before the reader
   call; fails otherwise.

7. **Failing test (DoD #4, no-filter policy enforced by the use case):**
   `rejectsCriteria_whenNoSubstantiveFilterIsPresent`. Build a criteria
   record with no substantive filter (if T-5 forbids this at
   construction time, build through whatever factory T-5 exposes — the
   point is to test the use-case's call to `validator.validate`). Call
   `execute(criteria, new RequestedLimit(50), Optional.empty())`. Assert:
   - throws `InvalidQueryException` with `ProblemKind.NO_FILTER`.
   - `fake.callCount == 0`.
   Run — green if step 3 puts the validator call first.
   *Open coordination point:* if T-5 collapses "no substantive filter"
   into a constructor-time invariant (so a criteria object cannot
   exist without a filter), this test instead asserts that the
   use case does not need an extra runtime check — covered by the
   "criteria-mapping" test of step 2. Resolve in review with T-5's
   author; the plan ships with a use-case-side check by default since
   `tasks.md` T-5 places policy in the use case.

8. **Cursor-decode failure passes through (sanity, not in DoD but
   keeps the contract honest):**
   `propagatesInvalidCursorException_fromCodec_withoutCallingReader`.
   Pass a literal `"!!not-base64!!"` cursor; assert
   `InvalidCursorException` with `ProblemKind.INVALID_CURSOR` bubbles
   up untouched and the reader was not called. ~10 LOC; cheap and
   protects the layering.

9. **Static checks.** `mvn spotless:apply` then `./mvnw test`.
   ArchUnit `LayeringTest` must stay green — the use case lives in
   `application/query/` and imports only `domain/shared/` +
   `application/query/*` (the validator, codec, and exceptions from
   T-5/T-6 also live in `application/query/` per their plans). No
   `org.springframework.web.*` or `jakarta.persistence.*` imports.

10. **LOC budget check.** `git diff --stat main...HEAD` excluding
    generated. Budget ~330 LOC; tests dominate the diff. If approaching
    400, the cheapest cut is to fold steps 5's two sub-cases into a
    single `@ParameterizedTest`.

## Risks / open questions

- **Glossary alignment.** Names, signatures, exception structure, and
  execution order in this plan are bound to
  [../glossary.md](../glossary.md) §§2, 3, 4, 7. Specifically:
  `QueryCriteria` does **not** carry `limit` — `RequestedLimit` is a
  separate parameter on `execute`; `EventReader.read` takes
  `Optional<KeysetPosition>` (not nullable);
  `KeysetPosition(OffsetDateTime recordedAt, UUID id)` and
  `QueryPage(List<AuditEvent> items, Optional<String> nextCursor)` are
  the exact record shapes; cursor decoding is a single
  `codec.decode(raw, expected)` call that both decodes and verifies
  fingerprint binding (no separate `verifyFingerprint`); T-6 ships
  one `InvalidCursorException` with nested
  `ProblemKind { INVALID_CURSOR, CURSOR_FILTER_MISMATCH }` (the
  earlier `CursorFilterMismatchException` is collapsed in); T-5 ships
  one `InvalidQueryException` with nested
  `ProblemKind { NO_FILTER, BLANK_FILTER, INVALID_TIME_RANGE }` (exact
  spelling `ProblemKind`, not `Kind`). If the glossary changes, this
  plan changes; the glossary wins.
- **`KeysetPosition` location.** Lives in `application/query/` here so
  it does not leak to `domain/`. T-8 imports it. If a future
  non-Postgres reader joins, the type stays put.
- **Order of validation vs cursor decode.** Plan validates criteria
  first, then decodes the cursor. Rationale: a request that violates
  the no-filter rule should fail the same way regardless of whether a
  cursor was supplied; checking the cursor first would surface an
  `invalid-cursor` 400 ahead of the `no-filter` 400, masking the real
  problem.
- **Empty result set encoding.** When the reader returns `0` rows,
  `nextCursor` is empty by the split. Matches design §2: "`next_cursor`
  is `null` on the last page (also when `items` is empty)."
- **Race with concurrent ingestion.** Out of scope here — design §5
  invariant 2 is enforced by the reader's strict-less-than predicate
  in T-8, not by the use case. The use case only needs to pass the
  correct keyset position through.
