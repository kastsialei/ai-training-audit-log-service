**Refs:** US-1, US-2, US-3 ([requirements.md](../requirements.md)), [design.md](../design.md) section 3 "New indexes" and "Read patterns covered"
**Depends on:** —

## Goal

Add the four composite indexes that make the canonical query patterns
viable, **before** the read adapter (T-8) starts depending on them. After
this task, Flyway carries migrations V3–V6 that create
`idx_audit_events_recorded_at_id`,
`idx_audit_events_actor_recorded_at_id`,
`idx_audit_events_resource_recorded_at_id`, and
`idx_audit_events_event_type_recorded_at_id` on `audit_events` using
plain transactional `CREATE INDEX`. The table is created in V2 and is
empty when V3–V6 run, both in Testcontainers and on first production
deploy — there is no ingestion to block. A focused integration test
proves Flyway applied the migrations and the four index names exist in
the catalog. No application code, no `outcome` index, no edits to V1/V2.

## Approach

- **Index set.** Exactly the four names in `design.md` §3 — no
  `outcome` variants, no partial indexes. Each composite is
  `(filter, recorded_at DESC, id DESC)` so the planner can serve the
  canonical `WHERE filter=? AND (recorded_at,id) < (?,?) ORDER BY
  recorded_at DESC, id DESC` pattern from a single index. The
  filter-free index `(recorded_at DESC, id DESC)` covers time-only
  queries (`from`/`to` only) and is also the fallback for the
  `outcome`-only case (recheck on `outcome`).
- **Transactional plain `CREATE INDEX`.** The table is created in V2
  and is empty when V3–V6 run, so blocking ingestion is a non-issue —
  plain `CREATE INDEX` inside Flyway's default transaction is correct
  and fastest. `CONCURRENTLY` is intentionally **not** used here: in
  Spring Boot + HikariCP startup it can hang on
  `WaitForOlderSnapshots` waiting for other idle pool connections,
  and it adds nothing for an empty table. Any later index against the
  populated production table belongs in a separate future migration
  that opts out of the transaction and uses `CONCURRENTLY`.
- **One index per file.** V3, V4, V5, V6 each create one index. This
  keeps each migration trivially reversible and makes Flyway's history
  table easy to read. No `IF NOT EXISTS` — these are the migrations
  that introduce the names; Flyway's history table guards
  re-application.
- **Naming.** Exactly the names from `design.md` §3, byte-for-byte —
  these names are part of the contract the integration test asserts on,
  and T-8's `EXPLAIN` test (per `design.md` §11) will refer to
  `idx_audit_events_actor_recorded_at_id` by literal name.
- **Verification approach.** A new `*IT.java` integration test boots
  Spring Boot + Testcontainers Postgres exactly like `HealthIT` /
  `IngestionIT`. By the time the Spring context is up, Flyway has run
  V1 → V2 → V3 → V4 → V5 → V6, so the test only needs to query
  `pg_indexes` via a `JdbcTemplate` and assert the four expected names
  are present on `audit_events`. Querying `pg_indexes` is simpler than
  `pg_class` + `pg_index` joins and avoids
  `information_schema.statistics` (which is a MySQL-flavoured view, not
  the canonical Postgres one).
- **Failing-first ordering.** Per AGENTS.md Test-driven rule, the
  test file lands first asserting the four index names exist. Run —
  it fails because V3–V6 don't exist yet. Then add V3–V6. Re-run — green.

## Risks / open questions

- **Why not `CONCURRENTLY`?** Initially the plan mandated
  `CONCURRENTLY` + `-- flyway:executeInTransaction=false`. In practice
  this hung Spring Boot startup: Postgres' `WaitForOlderSnapshots`
  waited on HikariCP's idle pool connections that hold a snapshot from
  earlier context-init work. Since V2 creates the table in the same
  migration series, V3–V6 run against an empty table — there is no
  ingestion to block, so `CONCURRENTLY` adds risk without benefit.
  Any later index on the populated production table belongs in a
  separate future migration with the opt-out directive.
- **Testcontainers Postgres image.** Codebase already uses
  `postgres:16-alpine`. Plain `CREATE INDEX` is universal across
  Postgres versions; no version risk. Reuse the same image string to
  avoid drift.
- **Index naming drift.** The four literal names are referenced by
  T-8's `EXPLAIN` test (per `design.md` §11). If anyone renames them
  here, T-8's test breaks. No action needed in T-4 beyond using the
  exact strings from `design.md` §3.
- **Empty-table index build.** Testcontainers' `audit_events` is empty
  when V3–V6 run; index build is instant. On the first production
  deploy the table is also empty, so the same is true. Any later
  index against a populated table belongs in a separate later migration.
- **`outcome` indexes (out of scope).** `design.md` §3 deliberately
  omits them; `design.md` §9 records the rejected alternative.
  Resisting scope creep is the only "risk" — flagged here so the
  reviewer doesn't request them.
