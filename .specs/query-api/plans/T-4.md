## Goal

Add the four composite indexes that make the canonical query patterns
viable, **before** the read adapter (T-8) starts depending on them. After
this task, Flyway carries a V3 migration that creates
`idx_audit_events_recorded_at_id`,
`idx_audit_events_actor_recorded_at_id`,
`idx_audit_events_resource_recorded_at_id`, and
`idx_audit_events_event_type_recorded_at_id` on `audit_events`. The
migration uses `CREATE INDEX CONCURRENTLY` and opts out of Flyway's
default transaction so it does not block ingestion. A focused
integration test proves Flyway applied the migration and the four index
names exist in the catalog. No application code, no `outcome` index, no
edits to V1/V2.

## Approach

- **Index set.** Exactly the four names in `design.md` §3 — no
  `outcome` variants, no partial indexes. Each composite is
  `(filter, recorded_at DESC, id DESC)` so the planner can serve the
  canonical `WHERE filter=? AND (recorded_at,id) < (?,?) ORDER BY
  recorded_at DESC, id DESC` pattern from a single index. The
  filter-free index `(recorded_at DESC, id DESC)` covers time-only
  queries (`from`/`to` only) and is also the fallback for the
  `outcome`-only case (recheck on `outcome`).
- **Concurrent creation.** Flyway wraps every migration in a
  transaction by default. `CREATE INDEX CONCURRENTLY` is **forbidden**
  inside a transaction by Postgres. The migration therefore opens with
  the Flyway-specific opt-out comment on its **own first line**:
  `-- flyway:executeInTransaction=false`. Without it, Flyway fails with
  `CREATE INDEX CONCURRENTLY cannot run inside a transaction block`.
  This is the only real gotcha in T-4.
- **Statement order.** Each `CREATE INDEX CONCURRENTLY` is its own
  statement, separated by a blank line. No `BEGIN`/`COMMIT`. No
  `IF NOT EXISTS` — V3 is the migration that introduces these names;
  if it has already run on a target DB, Flyway's history table guards
  re-application.
- **Naming.** Exactly the names from `design.md` §3, byte-for-byte —
  these names are part of the contract the integration test asserts on,
  and T-8's `EXPLAIN` test (per `design.md` §11) will refer to
  `idx_audit_events_actor_recorded_at_id` by literal name.
- **Verification approach.** A new `*IT.java` integration test boots
  Spring Boot + Testcontainers Postgres exactly like `HealthIT` /
  `IngestionIT`. By the time the Spring context is up, Flyway has run
  V1 → V2 → V3, so the test only needs to query `pg_indexes` via a
  `JdbcTemplate` and assert the four expected names are present on
  `audit_events`. Querying `pg_indexes` is simpler than `pg_class` +
  `pg_index` joins and avoids `information_schema.statistics` (which is
  a MySQL-flavoured view, not the canonical Postgres one).
- **Testcontainers + `CONCURRENTLY`.** The container exposes a normal
  Postgres connection; Flyway's `executeInTransaction=false` opens an
  auto-commit connection for V3, so `CONCURRENTLY` works the same as
  in any other Postgres environment. No special container configuration
  is required.
- **Failing-first ordering.** Per AGENTS.md Test-driven rule, the
  test file lands first asserting the four index names exist. Run —
  it fails because V3 doesn't exist yet. Then add V3. Re-run — green.

   `V3__create_audit_events_query_indexes.sql`. First line is exactly

## Risks / open questions

- **`CONCURRENTLY` + Flyway transaction default.** The single real
  failure mode. Mitigated by the opt-out directive on line 1 of V3 and
  the integration test that fails immediately if Flyway can't apply.
  Leading comment in V3 calls out the gotcha for future editors.
- **Testcontainers Postgres image.** Codebase already uses
  `postgres:16-alpine`. `CREATE INDEX CONCURRENTLY` has been stable
  since Postgres 9; no version risk. Reuse the same image string to
  avoid drift.
- **Index naming drift.** The four literal names are referenced by
  T-8's `EXPLAIN` test (per `design.md` §11). If anyone renames them
  here, T-8's test breaks. No action needed in T-4 beyond using the
  exact strings from `design.md` §3.
- **Empty-table index build.** Testcontainers' `audit_events` is empty
  when V3 runs; index build is instant. In production on a ~10M-row
  table the build is ~1–3 minutes per index (per `design.md` §3) but
  non-blocking thanks to `CONCURRENTLY`. No code change needed for
  either case.
- **`outcome` indexes (out of scope).** `design.md` §3 deliberately
  omits them; `design.md` §9 records the rejected alternative.
  Resisting scope creep is the only "risk" — flagged here so the
  reviewer doesn't request them.
