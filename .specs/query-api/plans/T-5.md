# T-5: Model Query Filters and Validation Policy

**Refs:** US-1, US-2, US-4 ([requirements.md](../requirements.md)), [design.md](../design.md) section 4 "Validation rules", section 4 "Where rules live", section 6 "Edge cases & failure modes"
**Glossary:** [../glossary.md](../glossary.md) — canonical names, types, exceptions, problem types
**Depends on:** T-3 (shared type promotion — `Outcome` must live in `domain/shared/`)
**Est. LOC:** ~260

## Goal

Stand up the read-side application model for query filters and the
plain-Java validation policy that owns the rules `design.md` §4 assigns
to "the use case", *without* introducing any HTTP, Spring Web, or DB
dependency. After this task, the project has a `QueryCriteria` value
record (and its sub-records for time bounds and requested limit) plus a
`QueryCriteriaValidator` that throws a typed, machine-classifiable
exception. T-7 will invoke the validator from the use case; T-10 will
map the exception to RFC 7807 problem types without string-parsing the
message.

## Approach

- New package `application/query/` for the read-side use-case-owned
  policy. Mirrors `application/ingestion/` (records + use case class,
  port living next to use case). Per ADR-0003, the package is created
  on first use — this task is that moment.
- Value records (Java `record`s, no Spring/Web/JPA imports):
  - `QueryCriteria` — the canonical bundle of substantive filters.
    Per glossary §2, fields are **exactly**:
    `String actor, String resource, String eventType, Outcome outcome, TimeRange timeRange`.
    All nullable (each individually optional). **`QueryCriteria` does
    NOT include `limit`** — `RequestedLimit` is a peer parameter, not
    a field on the criteria (glossary §2 note: keeps the cursor
    fingerprint scoped to substantive filters only). Raw strings are
    stored as-given; blank-after-trim rejection lives in the validator
    (see below).
  - `TimeRange(OffsetDateTime from, OffsetDateTime to)` — **both
    fields nullable**; `null` means unbounded on that side (glossary
    §2). The validator enforces `from <= to` when both are present.
    Uses `java.time.OffsetDateTime` per `design.md` §6 (UTC
    convention; offset honored on input).
  - `RequestedLimit(int value)` — a **peer of `QueryCriteria`, not a
    field on it**. Thin wrapper that gives T-7's use case method a
    typed slot for `limit` while keeping `QueryCriteria` clean
    (glossary §2). Range check `1..200` is **not** in this task —
    `design.md` §4 assigns the range check to the DTO (`@Min`/`@Max`)
    and the use case only cares that a parsed int reached it. We
    still model `RequestedLimit` here so the use case signature in
    T-7 stays clean.
- Validation policy — `QueryCriteriaValidator` (single static method
  `validate(QueryCriteria)` or instance method; pick instance for
  testability and to match `IngestEventUseCase` style of plain Java
  collaborators). Rules owned here, all per `design.md` §4
  "Where rules live":
  1. **≥1 substantive filter present** (`actor`, `resource`,
     `eventType`, `outcome`, `from`, `to`). `limit`/`cursor` do not
     count — they are not on `QueryCriteria` at all, which structurally
     encodes the rule.
  2. **Blank-after-trim** for `actor` / `resource` / `eventType`. If
     the field is non-null and `value.trim().isEmpty()`, reject.
     Per `design.md` §6: trim applies *only* to the blank check;
     matching itself is byte-exact, so the record still carries the
     raw value.
  3. **`from > to` rejected**; `from == to` permitted (half-open
     window — design §6).
- Typed exception — `InvalidQueryException` in `application/query/`
  (rationale: application-layer policy violation, not a domain
  invariant; mirrors the layering of the existing
  `InvalidAuditEventException` which lives next to the rule it
  enforces). Carries a **stable nested enum `ProblemKind`** so T-10
  maps it without string parsing. Signature exactly per glossary §4:
  ```java
  public final class InvalidQueryException extends RuntimeException {
      public enum ProblemKind { NO_FILTER, BLANK_FILTER, INVALID_TIME_RANGE }
      public ProblemKind kind();
      public Optional<String> field();   // present for BLANK_FILTER (which field was blank)
  }
  ```
  Per glossary §4, messages are **static, fixed strings** chosen by
  `ProblemKind` in the constructor and never include user-supplied
  input. `field()` returns the offending field name (`actor` /
  `resource` / `event_type`) only for `BLANK_FILTER`; for the other
  kinds it returns `Optional.empty()`.
- No `jakarta.validation`, no `@AssertTrue`, no `@NotBlank`, no
  `@Min`/`@Max` in this slice — those live on the DTO in T-9.
  ArchUnit `LayeringTest` already disallows `org.springframework.web`
  from `application/`; we additionally avoid `jakarta.validation` here
  to keep the policy a pure Java unit.

### Soft dependency on T-3

`QueryCriteria.outcome` references the shared `Outcome` enum. Today
`Outcome` lives in `domain/ingestion/`; T-3 promotes it to
`domain/shared/`. T-5 cannot land before T-3 — the plan assumes T-3
is merged. If T-3 slips, T-5 holds; we do not import the ingestion
package as a workaround (design §9 alternatives explicitly rejects
cross-feature imports).

## Step-by-step execution

Test-first per Ground rule 6.

1. **Failing test (DoD: no-filter rejection):** in
   `QueryCriteriaValidatorTest`, build a `QueryCriteria` with all
   substantive fields null (including `timeRange` carrying two
   `null`s, plus a `TimeRange` with both ends null — both cases). Call
   `validator.validate(criteria)`. Assert it throws
   `InvalidQueryException` whose `kind() == ProblemKind.NO_FILTER`.
   Run — fails (class does not yet exist).
2. **Implement `QueryCriteria`, `TimeRange`, `RequestedLimit`, `InvalidQueryException`, `QueryCriteriaValidator`** with just the NO_FILTER rule. Re-run — green.
3. **Failing test (DoD: blank `actor` rejection):** parameterized test
   over inputs `""`, `"   "`, `"\t"` for `actor`. Assert
   `InvalidQueryException` with `kind() == ProblemKind.BLANK_FILTER`
   and `field() == Optional.of("actor")`. Run — fails.
4. **Implement blank-after-trim rule** in the validator for `actor`.
   Re-run — green. Then repeat steps 3–4 for `resource` and
   `eventType` (same logic, three branches; keep DRY via a small
   private helper). Tests parameterized over `{actor, resource, event_type}`.
5. **Failing test (DoD: `from > to` rejection):** build criteria with
   `from = 2026-05-02T00:00:00Z`, `to = 2026-05-01T00:00:00Z`. Assert
   `InvalidQueryException` with
   `kind() == ProblemKind.INVALID_TIME_RANGE`. Run — fails.
6. **Implement `from > to` check** (`from.isAfter(to)`). Re-run — green.
7. **Failing test (DoD: `from == to` permitted):** build criteria with
   `from = to = 2026-05-01T00:00:00Z` and `actor = "u_42"` (so the
   no-filter rule is satisfied). Assert `validator.validate(criteria)`
   does **not** throw. This pins the half-open invariant from
   `design.md` §6 ("from == to → permitted"). Run — should already be
   green from step 6's `isAfter` (not `compareTo > 0`); if not, fix.
8. **Failing tests (DoD: independently optional `from` / `to`):** two
   tests — (a) only `from` set + a substantive filter → green; (b)
   only `to` set + a substantive filter → green. Both call
   `validate` and assert no throw. Run — green (the existing
   implementation only complains when both bounds are present and
   reversed). If a test fails, fix the null-guard in the time-range
   check.
9. **Failing test (DoD: `timeRange` alone satisfies "≥1 filter"):**
   only `from` (or only `to`) set, everything else null → validate
   does not throw. Confirms the timeRange counts as a substantive
   filter per `requirements.md` US-1 AC ("each `from` and `to` is
   independently optional") and `design.md` §2 footnote ¹.
10. **Failing test (DoD: stable error code accessible without parsing
    message):** assert that for each rejection case above,
    `ex.kind()` returns the right enum constant *and*
    `ProblemKind.values()` contains exactly `{NO_FILTER,
    BLANK_FILTER, INVALID_TIME_RANGE}`. This is the contract T-10
    will rely on; if a future rule lands, the enum is the source of
    truth.
11. **Static checks:** `mvn spotless:apply` then `./mvnw test`.
    ArchUnit `LayeringTest` must stay green — the new package depends
    only on `domain/shared/` (`Outcome`) and `java.time` /
    `java.util`. Explicitly verify no import of
    `org.springframework.*`, `jakarta.validation.*`, or
    `jakarta.persistence.*` in any of the new production files
    (`grep` step before commit).
12. **LOC budget check:** `git diff --stat main...HEAD` — exclude
    generated. Budget ~260 LOC; split if approaching 400.

## Risks / open questions

- **Where does `InvalidQueryException` live?** Picked
  `application/query/` over `domain/query/` because the rule is a
  use-case policy (`design.md` §4 "Where rules live" explicitly says
  the ≥1-filter rule lives in the use case, not the DTO and not a
  domain invariant). Layering stays clean: `api/` (T-10) may depend
  on `application/`. If a reviewer prefers `domain/query/`, the move
  is mechanical — no behavior change. Flag in PR.
- **`ProblemKind` enum vs. URI constants.** Chose enum so the
  exception is self-contained and the API layer never imports an
  application-layer string. T-10 will own the `ProblemKind → URI`
  mapping table inside `api/shared/ProblemTypes`. Alternative: store
  the URI on the exception itself. Rejected because that leaks API
  concerns into application/.
- **`RequestedLimit` placement.** `design.md` §4 puts limit-range
  validation on the DTO (`@Min(1)`/`@Max(200)`), so T-5 does *not*
  validate the limit value. `RequestedLimit` exists here only so
  T-7's use case method signature reads cleanly
  (`execute(QueryCriteria, RequestedLimit, Optional<Cursor>)`).
  If a reviewer pushes back, we can defer `RequestedLimit` to T-7;
  budget impact ~15 LOC.
- **Whitespace-only trim policy.** Implemented `value.trim().isEmpty()`
  (not `value.isBlank()`) because `design.md` §6 explicitly uses the
  word "trim" — semantically identical for ASCII whitespace, but
  picking `trim()` keeps the spec wording one-for-one. Either reads
  fine; reviewer can swap.
- **`TimeRange` with both ends null.** Today the validator treats
  this as "no time filter present" and contributes nothing to the
  "≥1 substantive filter" count. Confirmed against `design.md` §2
  footnote ¹ ("from, to" each count as substantive). A `TimeRange`
  record where both fields are null is structurally equivalent to no
  time filter; the validator dereferences `timeRange.from()` /
  `timeRange.to()` only after null-checking the wrapper.
- **Glossary alignment.** This plan adopts
  [`glossary.md`](../glossary.md) as the source of truth for type
  names, packages, and exception shape. Any drift between plan and
  glossary is a glossary update or a plan fix — never silent
  divergence.
