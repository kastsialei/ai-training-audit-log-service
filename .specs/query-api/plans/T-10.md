# T-10: Complete Query Validation Error Contract

**Refs:** US-3, US-4 ([requirements.md](../requirements.md)), [design.md](../design.md) section 4 "Validation rules", section 6 "Edge cases", section 7 "Integration points"
**Glossary:** [../glossary.md](../glossary.md) — canonical names, types, exceptions, problem types
**Depends on:** T-1 (global advice + `ProblemTypes`), T-6 (cursor codec + `InvalidCursorException` with nested `ProblemKind`), T-9 (controller, DTO, parameter binding)
**Est. LOC:** ~340

## Goal

Close out the query endpoint's 400 contract. After this task, every
client-input failure on `GET /audit-events` returns
`application/problem+json` with a **stable, feature-namespaced** `type`
URI from the eight slots enumerated in `design.md` §4 (`no-filter`,
`blank-filter`, `invalid-outcome`, `invalid-timestamp`,
`invalid-time-range`, `invalid-limit`, `invalid-cursor`,
`cursor-filter-mismatch`), all five RFC 7807 fields, and
`X-Correlation-Id`. Cursor errors never echo cursor bytes or decoded
JSON fields. T-9 already wires the controller and the success path;
T-1 already installs the `@RestControllerAdvice` and a generic
`validation-error` type; T-5 defines `InvalidQueryException` and T-6
defines `InvalidCursorException`. T-10 is **strictly the validation
finishing pass** — it extends the existing advice with two new
domain-aware handlers and refines two existing framework handlers so
they pick the right `type` based on the failing parameter.

## Approach

- Extend, do not duplicate. T-1's `GlobalExceptionHandler` and
  `ProblemTypes` already exist. T-10 adds two `@ExceptionHandler`
  methods and one shared mapping helper. No second advice class.
- Add eight new constants to `ProblemTypes` matching glossary §5 one
  for one — names and slugs MUST match exactly. Keep them under
  `https://audit-log-service/problems/`:
  - `NO_FILTER` → `no-filter`
  - `BLANK_FILTER` → `blank-filter`
  - `INVALID_OUTCOME` → `invalid-outcome`
  - `INVALID_TIMESTAMP` → `invalid-timestamp`
  - `INVALID_TIME_RANGE` → `invalid-time-range`
  - `INVALID_LIMIT` → `invalid-limit`
  - `INVALID_CURSOR` → `invalid-cursor`
  - `CURSOR_FILTER_MISMATCH` → `cursor-filter-mismatch`
- Two new handlers on `GlobalExceptionHandler`:
  - `InvalidQueryException` → `400`, `type` resolved from the
    exception's `ProblemKind` enum (defined in T-5, values
    `NO_FILTER`, `BLANK_FILTER`, `INVALID_TIME_RANGE` — three cases
    only). A small 3-case `switch` maps `ProblemKind` →
    `ProblemTypes.*` constant: `NO_FILTER` → `no-filter`,
    `BLANK_FILTER` → `blank-filter`, `INVALID_TIME_RANGE` →
    `invalid-time-range`. `INVALID_LIMIT` is **not** on this enum —
    limit-range violations come through `MethodArgumentNotValidException`
    (DTO `@Min`/`@Max`) and are routed to `invalid-limit` by the
    `MethodArgumentNotValidException` handler's field-name switch
    (refined here from T-1). `detail` comes from the exception's safe
    static message (T-5 guarantees no user input is interpolated).
  - `InvalidCursorException` → `400`. `InvalidCursorException` is
    **one class** with a nested `ProblemKind` enum
    (`INVALID_CURSOR`, `CURSOR_FILTER_MISMATCH`) per glossary §4. The
    handler dispatches on `ex.kind()`:

    ```java
    @ExceptionHandler(InvalidCursorException.class)
    ResponseEntity<ProblemDetail> handle(InvalidCursorException ex) {
        String type = switch (ex.kind()) {
            case INVALID_CURSOR -> ProblemTypes.INVALID_CURSOR;
            case CURSOR_FILTER_MISMATCH -> ProblemTypes.CURSOR_FILTER_MISMATCH;
        };
        // detail is a constant per kind — never ex.getMessage() echoing user input
    }
    ```

    There is no separate `CursorFilterMismatchException` class.
    `detail` is drawn from a **constant string table** keyed by
    `ProblemKind`, never from the exception message or the supplied
    cursor.
- Refine two existing framework handlers from T-1:
  - `MethodArgumentTypeMismatchException` — T-1 maps this to a
    generic `validation-error`. T-10 inspects `ex.getName()`:
    - `outcome` → `invalid-outcome`
    - `from` or `to` → `invalid-timestamp`
    - other → keep T-1's generic `validation-error` (so ingestion
      and future endpoints stay unaffected).
  - `MethodArgumentNotValidException` — T-1 maps this generically.
    T-10 inspects the failing field path:
    - `limit` → `invalid-limit`
    - `actor` / `resource` / `eventType` → `blank-filter`
    - cross-field `fromBeforeTo` (the class-level `@AssertTrue`
      method name) → `invalid-time-range`
    - other → keep T-1's generic `validation-error`.

    Field-name matching is acceptable here because the field names
    are stable contracts (design §2 + glossary §6.1).
- DTO-level validation lives on `api.query.QueryAuditEventsRequest`
  (T-9 ships the skeleton record per glossary §6.1; T-10 adds the
  validation annotations):
  - `@Min(1) @Max(200) Integer limit` →
    `MethodArgumentNotValidException` → routed to `invalid-limit` by
    the field-name switch above.
  - `@AssertTrue` instance methods on the record for blank
    `actor` / `resource` / `eventType` (trim-before-check, per
    design §6 "blank after trim") → routed to `blank-filter` by the
    field-name switch above.
  - `@AssertTrue` cross-field method for `from <= to` (when both
    present) → routed to `invalid-time-range`.

  Note: `blank-filter` and `invalid-time-range` have **two sources**
  — DTO-level (`@AssertTrue` via `MethodArgumentNotValidException`)
  and use-case-level (`InvalidQueryException` thrown by
  `QueryCriteriaValidator` in T-5). Both map to the same problem
  URIs. The DTO-level catch is the primary path; the use-case-level
  catch is a defense-in-depth in case the DTO is bypassed (e.g. in
  unit tests that call the use case directly).
- Timestamp without offset: T-9 binds `from` / `to` as
  `OffsetDateTime`. Spring's default conversion **does** reject
  no-offset strings via DateTimeException, surfacing as
  `MethodArgumentTypeMismatchException`. If empirical check during
  step 4 shows Spring silently widens `LocalDateTime` to a default
  offset, register a tiny `Converter<String, OffsetDateTime>` in
  `api/shared/` that calls `OffsetDateTime.parse(...)` strictly. The
  plan keeps the converter as a contingency rather than a default
  add — verify first, add only if needed (Simplicity first).
- "No-filter" rule ordering (design §4 + T-7): the use case rejects
  the filter set **before** decoding the cursor. So `?cursor=xxx`
  alone trips `no-filter`, not `invalid-cursor`. Tests assert this
  ordering — it's a property of the use case wired by T-7/T-9, but
  T-10 owns the test that locks it in at the HTTP level.
- Cursor-internal leak guarantee: the response `detail` is a constant
  from a string table (e.g. `"cursor is malformed"`,
  `"cursor was issued for a different filter set"`). A focused
  integration test exercises the **`cursor-filter-mismatch` path**:
  build a structurally valid cursor (valid base64, valid JSON, valid
  UUID `id`, hex `fp`, ISO-8601 `rt`) whose `rt` field carries a
  recognizable token like `"2099-12-31T23:59:59.000000001Z"`, then
  send it with filters whose fingerprint does not match. Assert the
  entire response body never contains that exact ISO substring.
  This proves the no-leak guarantee for the
  `cursor-filter-mismatch` path. A second test covers the
  `invalid-cursor` path with a malformed base64 input.
- `X-Correlation-Id` is set by T-2's filter before the chain runs, so
  it survives the advice path automatically. Every 400 test asserts
  the header is present.
- Parameterize aggressively. JUnit 5 `@ParameterizedTest` drives the
  eight problem types from a single `@CsvSource` (request → expected
  `type`) so the test class stays well under the LOC budget. Hand-
  written tests only where a parameterized case can't express the
  assertion (cursor leak guard, ordering of no-filter vs cursor).

## Step-by-step execution

Test-first per Ground rule 6.

1. **Failing test (DoD: no substantive filters):** in
   `QueryValidationIT`, send `GET /audit-events` with no query
   string. Assert `400`, `application/problem+json`, RFC 7807 fields
   present, `type` ends with `/no-filter`, `X-Correlation-Id`
   present. Run — fails (T-9 likely returns either `200`-empty or a
   generic `validation-error` depending on how `InvalidQueryException`
   surfaces; this test pins the contract).
2. **Implement the `InvalidQueryException` handler** on
   `GlobalExceptionHandler`. Add the new `ProblemTypes` constants
   first (all eight per glossary §5). Map `ProblemKind` → constant
   via a 3-case `switch` (`NO_FILTER` → `NO_FILTER`,
   `BLANK_FILTER` → `BLANK_FILTER`, `INVALID_TIME_RANGE` →
   `INVALID_TIME_RANGE`). Do **not** add an `INVALID_LIMIT` case —
   that path lives on the `MethodArgumentNotValidException` handler
   (step 10). Re-run step 1 — green.
3. **Failing tests (DoD: only `limit`, only `cursor`):** two
   parameterized cases — `?limit=10` → `no-filter`,
   `?cursor=anything` → `no-filter`. The cursor case locks in that
   `no-filter` fires **before** cursor decoding (design §4 + T-7
   ordering). Run — green if step 2 is correct and T-7 wired the
   ordering; if the cursor case returns `invalid-cursor`, the bug is
   in T-7's use case, not this advice — raise and fix there before
   merging T-10.
4. **Verify timestamp converter behavior.** Add a one-off
   exploratory test sending `?from=2026-05-01T00:00:00` (no offset)
   and `?actor=u_1`. Observe: does Spring reject (→
   `MethodArgumentTypeMismatchException`) or silently parse with a
   default offset? If reject: continue, no converter needed. If
   parse: add `StrictOffsetDateTimeConverter` and a
   `WebMvcConfigurer` to register it, retest.
5. **Failing tests (DoD: invalid timestamp, no-offset timestamp):**
   two cases — `?from=not-a-date&actor=u_1` and
   `?from=2026-05-01T00:00:00&actor=u_1`. Assert
   `type=…/invalid-timestamp`. Run — fails (T-1's generic
   `validation-error` is still being used).
6. **Refine the `MethodArgumentTypeMismatchException` handler.**
   Inspect `ex.getName()`: `from` / `to` → `INVALID_TIMESTAMP`;
   `outcome` → `INVALID_OUTCOME`; else keep existing
   `validation-error`. Re-run step 5 — green.
7. **Failing test (DoD: invalid `outcome`):** parameterized case
   `?outcome=BOGUS&actor=u_1` → `type=…/invalid-outcome`. Run —
   green if step 6 covered it.
8. **Failing tests (DoD: blank filters, whitespace-only filters):**
   four parameterized cases — `?actor=&resource=u_1`,
   `?resource=&actor=u_1`, `?event_type=&actor=u_1`,
   `?actor=%20%20&resource=u_1`. Each → `type=…/blank-filter`.
   Run — fails (T-9's DTO has no blank validators).
9. **Add DTO `@AssertTrue` blank validators** to the T-9 request
   object (`isActorNonBlankIfPresent` etc., each trimming before
   testing emptiness). Re-run step 8 — fails next, because
   `MethodArgumentNotValidException` maps to generic
   `validation-error`.
10. **Refine the `MethodArgumentNotValidException` handler** to read
    the first field error's path: `actor`/`resource`/`eventType` →
    `BLANK_FILTER`; `limit` → `INVALID_LIMIT`; the cross-field method
    `fromBeforeOrEqualTo` (class-level error) → `INVALID_TIME_RANGE`.
    Fallback `validation-error`. Re-run step 8 — green.
11. **Failing tests (DoD: `from > to`, `limit < 1`, `limit > 200`):**
    three parameterized cases. `from > to` → `invalid-time-range`;
    `limit=0` and `limit=201` → `invalid-limit`. Run — fails (no
    DTO validators for those yet).
11a. **Failing unit test (DoD: pin DTO `limit` component name for the
     field-name switch — glossary §5, design §4):** add a focused
     unit test
     `GlobalExceptionHandlerLimitFieldNameTest::limitComponentName_pinsInvalidLimitRouting`
     under `src/test/java/.../api/shared/`. The test uses
     reflection (or, more cheaply, a Spring `WebMvcTest` slice) to
     assert that the DTO `QueryAuditEventsRequest` exposes the
     component name **exactly** `"limit"` — the same literal the
     `MethodArgumentNotValidException` handler's field-name switch
     keys on. Rationale: a silent rename
     (`Integer limit` → `Integer maxResults`) would re-route limit
     failures to the generic `validation-error` and break the
     `INVALID_LIMIT` contract. This test fails immediately if the
     component is renamed without updating the switch table.
12. **Add `@Min(1) @Max(200)` on `limit`** and the `@AssertTrue
    fromBeforeOrEqualTo` cross-field method on the T-9 DTO. Re-run
    step 11 — green via the field-name mapping from step 10.
13. **Failing tests (DoD: malformed cursor, cursor/filter
    mismatch):** two cases — `?cursor=@@@not-base64@@@&actor=u_1`
    → `invalid-cursor`; a structurally valid cursor whose
    fingerprint doesn't match the supplied filters →
    `cursor-filter-mismatch`. Build the latter by encoding a valid
    cursor for filter set A, then sending it with filter set B. Run
    — fails (no advice handler for `InvalidCursorException` yet).
14. **Implement the `InvalidCursorException` handler.** Dispatch on
    `ex.kind()` (single-class with nested `ProblemKind` per
    glossary §4) to `INVALID_CURSOR` / `CURSOR_FILTER_MISMATCH`.
    `detail` is read from a constant table inside the handler —
    never from the exception message. Re-run step 13 — green.
15. **Failing test (DoD: cursor-internal leak guard,
    `cursor-filter-mismatch` path):** craft a **structurally valid**
    cursor — valid base64, valid JSON, valid UUID `id`, hex `fp`,
    ISO-8601 `rt` — where the `rt` field carries a recognizable
    token like `"2099-12-31T23:59:59.000000001Z"`. Send it with
    filters whose fingerprint does not match the cursor's `fp`.
    Assert response body does **not** contain the exact ISO
    substring `2099-12-31T23:59:59.000000001Z`, does not contain
    the raw cursor string, and `detail` equals the constant "cursor
    was issued for a different filter set". A previous draft used
    `LEAKED-MARKER` in `fp`, but that would fail format validation
    and route to `invalid-cursor` instead — the marker must ride in
    a structurally valid field so the request reaches the
    `cursor-filter-mismatch` branch. Add a **second** test for the
    `invalid-cursor` path that sends a malformed base64 input and
    asserts `type=…/invalid-cursor` plus a constant `detail`.
    Run — fails if the handler used `ex.getMessage()` accidentally;
    green when the constant table is in place.
16. **Failing test (DoD: every 400 carries `X-Correlation-Id`):** a
    parameterized assertion across **all** prior 400 cases checks
    header presence. Two flavors: (a) caller supplies
    `X-Correlation-Id: fixed-id` → asserted equal; (b) no header →
    UUID-shape match. Re-run — green; T-2's filter sets the header
    before the chain so it persists through the advice.
17. **Static checks:** `mvn spotless:apply`, then
    `./mvnw verify`. ArchUnit `LayeringTest` must stay green —
    the advice already lives in `api/shared/`; the new DTO
    validators live in `api/query/` (T-9 layer); no domain or
    infrastructure dependency is added.
18. **LOC budget check (DoD: <400 LOC):**
    `git diff --stat main...HEAD` excluding generated. Budget is
    ~340 LOC. The parameterized integration test is the largest
    file; if it pushes the diff toward 400, fold setup helpers into
    a single static `@BeforeAll` fixture rather than per-test setup.

## Risks / open questions

- **Spring's `OffsetDateTime` strictness.** Whether the default
  converter rejects no-offset input is environment-dependent (Spring
  6.x). Step 4 verifies empirically; if a strict converter is
  needed, it adds ~30 LOC. No design change either way — the
  observable contract is `400 invalid-timestamp`.
- **Glossary alignment.** This plan tracks
  [../glossary.md](../glossary.md) as the canonical source for
  exception shape, `ProblemKind` enums, problem-type slugs, and the
  T-9 request DTO. Specifically:
  `InvalidQueryException.ProblemKind` has exactly three values
  (`NO_FILTER`, `BLANK_FILTER`, `INVALID_TIME_RANGE`);
  `InvalidCursorException` is a single class with a nested
  `ProblemKind { INVALID_CURSOR, CURSOR_FILTER_MISMATCH }` (no
  separate `CursorFilterMismatchException`); `INVALID_LIMIT` is
  routed by the `MethodArgumentNotValidException` field-name switch,
  not by `InvalidQueryException`. If T-5/T-6 land with a different
  shape, raise against the glossary first — the glossary wins and
  the upstream task must be updated, not this plan.
- **Field-name string matching.** The
  `MethodArgumentNotValidException` and
  `MethodArgumentTypeMismatchException` mappings inspect parameter /
  field names by string. Field names are stable contracts per design
  §2, but if T-9 renames a DTO field (e.g. `eventType` vs
  `event_type`), the mapping must follow. Lock the names in T-9's
  DTO and reference them in a single private constant set on
  `GlobalExceptionHandler`.
- **Ordering of `no-filter` vs `invalid-cursor`.** Asserted by step
  3. If T-7's use case decodes the cursor before checking the
  filter set, the test will fail and the fix is in T-7, not here.
  Flag during T-10 review.
- **LOC headroom.** 340 is tight. If the parameterized matrix and
  the dedicated tests together push past 400, peel the leak-guard
  and correlation-ID-on-400 tests into a sibling `IT` file rather
  than relaxing assertions.
