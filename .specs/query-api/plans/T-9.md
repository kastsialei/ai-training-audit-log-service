# T-9: Expose GET /audit-events Success Path

**Refs:** US-1, US-2, US-3 ([requirements.md](../requirements.md)), [design.md](../design.md) section 2 "Public surface / contract"
**Glossary:** [../glossary.md](../glossary.md) — canonical names, types, exceptions, problem types
**Depends on:** T-1, T-2, T-7, T-8
**Est. LOC:** ~380

## Goal

Expose the read endpoint for *successful* queries. After this task,
`GET /audit-events` accepts the eight query parameters in `design.md`
§2, delegates to the T-7 use case, and returns the snake_case envelope
`{ "items": [...], "next_cursor": "..."|null }` with each item mapped
exactly as persisted (`id`, `recorded_at`, `actor`, `event_type`,
`resource`, `outcome`, `context`). Cross-cutting concerns (RFC 7807,
`X-Correlation-Id`) come in for free via T-1 and T-2. The full 400
matrix and pagination invariants are explicitly **not** in this slice
— they belong to T-10 and T-11.

## Approach

- New `api/query/` package (mirrors `api/ingestion/`). One controller +
  one request parameter object + two response DTO records. The request
  parameter object `QueryAuditEventsRequest` is a Spring
  `@ModelAttribute`-bound record (per glossary §6.1) so T-10 can attach
  `@AssertTrue`, `@Min`, `@Max` validators. T-9 ships the record
  skeleton **without** validation annotations; T-10 owns those.
- The controller method signature is exactly:
  ```java
  @GetMapping("/audit-events")
  public AuditEventQueryResponse query(@Valid @ModelAttribute QueryAuditEventsRequest request);
  ```
  `@Valid` is harmless here (no annotations on the record yet) and
  becomes load-bearing when T-10 lands.
- Controller is the *only* HTTP adapter: it builds a `QueryCriteria`
  (no `limit`/`cursor` — per glossary §2 note) from the request record,
  builds a `RequestedLimit` (defaulted to `50` when `request.limit()`
  is `null`), wraps `request.cursor()` as `Optional<String>`, calls
  `useCase.execute(criteria, limit, rawCursor)`, maps the resulting
  `QueryPage` (from T-7) into the response envelope, and returns it
  (Spring serializes to `application/json`).
- Snake_case serialization follows the existing ingestion convention:
  per-field `@JsonProperty("snake_name")` on the DTO records.
  `application.yml` is **not** changed — `IngestEventRequest` /
  `IngestEventResponse` already rely on the framework default
  (`LOWER_CAMEL_CASE`), so switching to global
  `property-naming-strategy=SNAKE_CASE` would silently break ingestion.
  Per-field annotations keep the change local to query DTOs.
- `context` is `JsonNode` end-to-end (matches `IngestEventResponse`):
  Jackson re-serializes the raw JSONB shape with no field reordering
  or normalization.
- The "cap page size to requested limit" rule is satisfied by T-7: the
  use case requests `limit + 1` from the reader and drops the extra
  item before returning. T-9 just trusts the result; no extra slicing.
- Default `limit=50` is applied by the **controller** when
  `request.limit()` is `null` (the record's `Integer limit` field
  stays nullable so a missing parameter is distinguishable from an
  explicit value). The use case receives a non-null `RequestedLimit`.
  Range validation (`1 ≤ limit ≤ 200`) is T-10's scope; we do not add
  `@Min`/`@Max` on the record here. A binder out-of-range value
  (e.g., `limit=201`) reaches the use case unchanged in T-9 — T-10
  will tighten it via DTO annotations.
- Snake_case URL param binding for `event_type` → `eventType`: per
  glossary §6.1, the binding mechanism is **pinned**: `@ModelAttribute`
  on the request record **plus** explicit `@RequestParam(name =
  "event_type")` on each snake_case → camelCase record component.
  Property-based naming strategies (e.g. setting a global
  `spring.mvc.servlet.path` / Jackson property strategy) are
  **rejected** because they would also affect ingestion DTOs. T-9
  applies the same per-component `@RequestParam(name = "...")`
  treatment to any other future snake_case binding (none today beyond
  `event_type`). A dedicated integration test pins the binding (see
  step-by-step) so a regression of the annotation strategy fails
  loudly.
- `outcome` is bound as the shared `Outcome` enum: Spring's default
  enum converter is case-sensitive uppercase, matching `design.md` §2.
  An invalid value raises `MethodArgumentTypeMismatchException` →
  caught by T-1's advice as `400`. T-10 swaps the `type` URI to
  `invalid-outcome`; this slice tolerates the generic mapping.
- `from`/`to` bind as `OffsetDateTime`. Spring converts ISO-8601 with
  offset out-of-the-box; missing offset or unparseable values produce
  `MethodArgumentTypeMismatchException` → handled identically.
- The `X-Correlation-Id` header is set by T-2's filter before
  `chain.doFilter`, so the success path inherits it. T-9 only asserts
  the header in tests — no production code.
- Unknown query parameters: accept-and-ignore is Spring's default. Per
  `requirements.md` §Out of scope, this is **not** a contract — no
  test asserts either acceptance or rejection.

## Files

**Created**

- `src/main/java/net/sam/ai/engineering/audit/api/query/QueryAuditEventsController.java`
  — `@RestController @RequestMapping("/audit-events")`, single
  `@GetMapping` method taking `@Valid @ModelAttribute QueryAuditEventsRequest request`,
  building `QueryCriteria` (no `limit`/`cursor`), `RequestedLimit`
  (defaulted to 50 if `request.limit()` is `null`), and
  `Optional<String> rawCursor`, invoking
  `useCase.execute(criteria, limit, rawCursor)`, mapping the result,
  returning `AuditEventQueryResponse`. ~70-90 LOC.
- `src/main/java/net/sam/ai/engineering/audit/api/query/QueryAuditEventsRequest.java`
  — record skeleton (no validation annotations; owned by T-10). Each
  snake_case → camelCase record component carries an explicit
  `@RequestParam(name = "…")` so the binding is pinned at the
  annotation level rather than via global naming strategy (glossary
  §6.1):
  ```java
  public record QueryAuditEventsRequest(
      String actor,
      String resource,
      @RequestParam(name = "event_type") String eventType,
      Outcome outcome,
      OffsetDateTime from,
      OffsetDateTime to,
      Integer limit,              // null → controller applies default 50
      String cursor
  ) {}
  ```
  ~25 LOC.
- `src/main/java/net/sam/ai/engineering/audit/api/query/AuditEventQueryResponse.java`
  — record:
  ```java
  public record AuditEventQueryResponse(
      List<AuditEventResponse> items,
      @JsonProperty("next_cursor") String nextCursor
  ) {}
  ```
  ~20 LOC.
- `src/main/java/net/sam/ai/engineering/audit/api/query/AuditEventResponse.java`
  — record matching `IngestEventResponse` field-for-field
  (`id`, `recorded_at`, `actor`, `event_type`, `resource`, `outcome`,
  `context`) with a static `from(AuditEvent)` mapper. ~35 LOC.
- `src/test/java/net/sam/ai/engineering/audit/api/query/QueryAuditEventsIT.java`
  — Testcontainers + MockMvc integration test class. Mirrors
  `IngestionIT` boot/datasource setup. Seeds rows directly via
  `JdbcTemplate` (or the persisted `AuditEventRepository`) so the
  read path is exercised without coupling to ingestion. ~180-220 LOC,
  six tests (see step-by-step).

**Modified**

- None. T-9 is purely additive in production code; existing tests in

## Step-by-step execution

Test-first per Ground rule 6. All tests in `QueryAuditEventsIT` unless
noted.

1. **Failing test (DoD #1):** `getFilteredQuery_returns200ApplicationJson`.
   Seed one event (`actor="u_42"`, fixed `recorded_at`). Call
   `GET /audit-events?actor=u_42`. Assert status `200`, `Content-Type`
   starts with `application/json`. Run — fails (no controller, 404).
2. **Implement the request and response DTOs**:
   `QueryAuditEventsRequest` (record skeleton, no validation
   annotations — T-10 owns those), `AuditEventResponse`, and
   `AuditEventQueryResponse`. Use per-field `@JsonProperty` for the
   snake_case response fields and a `from(AuditEvent)` mapper on the
   item DTO. No logic on the request record.
3. **Implement `QueryAuditEventsController`** with the success path
   only: take `@Valid @ModelAttribute QueryAuditEventsRequest request`,
   build a `QueryCriteria` from T-5's record (no `limit`/`cursor`),
   build `RequestedLimit` (defaulting to `50` when `request.limit()`
   is `null`), wrap `request.cursor()` as `Optional<String> rawCursor`,
   call `useCase.execute(criteria, limit, rawCursor)`, map
   `QueryPage.items()` → `List<AuditEventResponse>` via the static
   mapper, map `QueryPage.nextCursor()` → `String` via
   `.orElse(null)`, and return the
   `AuditEventQueryResponse` envelope. Re-run test 1 — green.
4. **Failing test (DoD #2 — envelope shape):**
   `response_usesSnakeCaseEnvelopeAndItemFields`. Seed one event with
   all fields populated. Call the endpoint. Assert via JSON path:
   `$.items` is an array of size 1, `$.next_cursor` exists (may be
   `null`), `$.items[0].id`, `$.items[0].recorded_at`,
   `$.items[0].actor`, `$.items[0].event_type`,
   `$.items[0].resource`, `$.items[0].outcome`,
   `$.items[0].context` are all present. Additionally parse the body
   as a `Map<String,Object>` and assert no key matches the camelCase
   forms `recordedAt`, `eventType`, `nextCursor` (regression guard for
   accidental global Jackson reconfig). Should pass after step 3 if
   `@JsonProperty` is right; fix DTOs if not.
5. **Failing test (DoD #3 — empty result):**
   `noMatches_returnsEmptyItemsAndNullCursor`. Seed nothing matching.
   Call `GET /audit-events?actor=does-not-exist`. Assert `200`,
   `$.items` length `0`, `$.next_cursor` is JSON `null` (use
   `jsonPath("$.next_cursor").value(nullValue())`). Should pass after
   step 3 — the use case returns an empty `QueryPage` with
   `nextCursor=null`.
6. **Failing test (DoD #4 — AND semantics over HTTP):**
   `multipleFilters_appliedAsAnd`. Seed three events:
   `(actor=u_1, resource=doc:1)`, `(actor=u_1, resource=doc:2)`,
   `(actor=u_2, resource=doc:1)`. Call
   `GET /audit-events?actor=u_1&resource=doc:1`. Assert `$.items`
   length `1` and that item is the `(u_1, doc:1)` row by `id`. This
   exercises the controller→use case→reader chain end-to-end with two
   filters. Should pass once T-8's reader applies the predicates.
7. **Failing test (DoD #5 — sort order):**
   `responseOrder_isRecordedAtDescThenIdDesc`. Seed four events with
   controlled `(recorded_at, id)` pairs: two share the latest
   `recorded_at` with different UUIDs, two share an older
   `recorded_at`. Call the endpoint with a single filter that matches
   all four. Assert `$.items[*].id` sequence is the expected
   `(recorded_at DESC, id DESC)` order — newest timestamp's
   larger-UUID first, then newest timestamp's smaller-UUID, then older
   timestamp's larger-UUID, then older timestamp's smaller-UUID.
   Tie-break correctness is T-8's contract; this test proves the
   ordering survives the HTTP layer unchanged.
8. **Failing test (DoD #6 — first page emits non-null `next_cursor`):**
   `firstPageOverLimit_emitsNonNullNextCursor`. Seed `limit+1` matching
   events (e.g., seed three, query `limit=2`). Assert `$.items` length
   `2` and `$.next_cursor` is a non-null, non-empty string. Do **not**
   walk the cursor — pagination invariants live in T-11.
9. **Failing test (DoD #7 — supplied correlation ID echoed):**
   `correlationId_echoedOnSuccessfulQuery`. Send the same call as test
   1 with header `X-Correlation-Id: fixed-uuid-like-value`. Assert
   response header `X-Correlation-Id` equals the supplied value.
   Should pass after T-2 merges; fails if T-2 isn't present yet —
   that's a dependency violation, not a T-9 bug.
10. **Failing test (DoD #7 — generated correlation ID present):**
    `correlationId_generatedWhenAbsentOnSuccessfulQuery`. Same call,
    no header. Assert response header `X-Correlation-Id` matches UUID
    regex `^[0-9a-fA-F]{8}-([0-9a-fA-F]{4}-){3}[0-9a-fA-F]{12}$`.
    Should pass after T-2.
11. **Failing test (DoD: snake_case binding mechanism pinned per
    glossary §6.1):** `eventType_bindsFromSnakeCaseRequestParam`.
    Seed one event with `event_type="doc.read"`. Call
    `GET /audit-events?event_type=doc.read` (snake_case in the URL).
    Assert `200` and `$.items` length `1` with
    `$.items[0].event_type == "doc.read"`. Regression guard: a second
    call to `GET /audit-events?eventType=doc.read` (camelCase) must
    **not** bind — assert the result is either empty (if other
    filters force a match) or a `400` (if Spring rejects the unknown
    name); the contract is that snake_case is the only accepted form,
    not that camelCase is also accepted. This test locks the
    `@RequestParam(name = "event_type")` strategy so a future
    refactor to a global naming strategy fails loudly.
12. **Static checks.** `mvn spotless:apply`, then `./mvnw test` and
    `./mvnw verify` (Testcontainers). ArchUnit `LayeringTest` must
    stay green — `api/query/` depends on `application/query/` (port)
    and `domain/shared/` (DTO mapping), both inward.
13. **LOC budget check.** `git diff --stat main...HEAD` excluding
    generated; budget ~380 LOC. If close to 400, split the test class
    (e.g., move envelope-shape + correlation-ID tests into a focused
    `QueryAuditEventsEnvelopeIT`) before opening the PR. Do **not**
    push the ceiling.

## Risks / open questions

- **Snake_case strategy collision with ingestion DTOs.** Resolved:
  ingestion uses per-field `@JsonProperty`; T-9 follows the same
  convention. Global Jackson naming strategy is *not* introduced.
  Regression guard for accidental future reconfig lives in the
  envelope-shape test (asserts absence of camelCase keys).
- **Dependency on T-3's `domain/shared/` move.** Imports in
  `AuditEventResponse.from(AuditEvent)` assume the post-T-3 location.
  If T-9 starts before T-3 merges, swap to `domain/ingestion/`
  temporarily and rebase. Mechanical, no logic impact.
- **T-7 result type shape.** Per glossary §2, `QueryPage` is exactly
  `record QueryPage(List<AuditEvent> items, Optional<String> nextCursor)`
  and the use case signature is exactly
  `QueryPage execute(QueryCriteria criteria, RequestedLimit limit, Optional<String> rawCursor)`.
  The controller maps `page.nextCursor().orElse(null)` into the
  response DTO's `String nextCursor` field. Any drift between this
  plan and the actual T-7 PR is resolved in favor of the glossary —
  the DTO contract does not change.
- **Snake_case binding mechanism for `event_type` (resolved).**
  Pinned per glossary §6.1: `@ModelAttribute` on the request record
  plus an explicit `@RequestParam(name = "event_type")` on the record
  component. No spike, no fallback. The
  `eventType_bindsFromSnakeCaseRequestParam` integration test locks
  the strategy so a future refactor (e.g. switching to a global
  naming strategy) fails loudly. Property-based naming strategies are
  explicitly rejected — they would also affect ingestion DTOs.
- **Glossary alignment.** This plan conforms to
  [../glossary.md](../glossary.md) v1 (2026-05-11): request parameter
  object `QueryAuditEventsRequest` adopted, use-case signature
  `execute(QueryCriteria, RequestedLimit, Optional<String>)`,
  `QueryPage` carries `Optional<String> nextCursor`, `Outcome` lives
  in `domain.shared`, response envelope keeps per-field
  `@JsonProperty` for snake_case. If the glossary changes, this plan
  must be re-reviewed before implementation.
- **Invalid `outcome` / `from` / `to` get a generic `validation-error`
  `type` URI today.** T-10 swaps in `invalid-outcome` /
  `invalid-timestamp`. No T-9 test asserts the specific URI for these
  cases (out of scope per `tasks.md`).
- **No `@Min`/`@Max` on `limit` here.** Out of range is silently
  forwarded; T-10 owns the mapping. Document in PR description so the
  reviewer doesn't flag this as missing validation.
- **`outcome=null` round-trip.** Persisted nullable per `design.md`
  §2. Jackson serializes `null` to JSON `null` by default; if the
  envelope-shape test seeds a row with `outcome=null`, the assertion
  uses `jsonPath("$.items[0].outcome").value(nullValue())`. No
  Jackson include-non-null config is added.
