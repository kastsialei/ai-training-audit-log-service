# T-2: Add Correlation-ID Filter

**Refs:** US-4 ([requirements.md](../requirements.md)), [design.md](../design.md) section 2 "Request headers", section 7 "Prerequisites"
**Depends on:** —
**Est. LOC:** ~220

## Goal

Install the cross-cutting request-correlation prerequisite the query
endpoint will inherit. After this task, every HTTP request — including
the existing `POST /audit-events` — has a correlation id bound to MDC
for the duration of the request and echoed on the response as
`X-Correlation-Id`. If the caller supplied the header, the same value
is echoed; if not, a UUID is generated. MDC is always cleaned up. No
query-endpoint code is added (T-9) and no metrics or request-summary
logging is added (out of scope per `requirements.md`).

## Approach

- New `OncePerRequestFilter` under `api/shared/`. `@Component` so
  Spring auto-registers it. `@Order(Ordered.HIGHEST_PRECEDENCE)` so the
  MDC key is populated before validation, binding, or the T-1
  `@RestControllerAdvice` runs.
- Header name and MDC key are constants in the filter class (no magic
  strings; one source of truth):
  - Header: `X-Correlation-Id` (matches `design.md` §2 exactly; no
    alternate spellings — per T-2 Notes in `tasks.md`).
  - MDC key: `correlationId` (matches `ARCHITECTURE.md` and the
    existing `logback-spring.xml` pattern `%X{correlationId:-}`).
- Algorithm in `doFilterInternal`:
  1. Read `X-Correlation-Id` from request.
  2. If absent or blank → generate `UUID.randomUUID().toString()`.
  3. `MDC.put("correlationId", id)`.
  4. `response.setHeader("X-Correlation-Id", id)` **before** calling
     `chain.doFilter(...)` — this guarantees the header survives both
     the success path and the error advice path. (`ProblemDetail` from
     T-1 replaces body + content-type only; pre-set headers persist.)
  5. `try { chain.doFilter(...); } finally { MDC.remove("correlationId"); }`
- Whitespace policy: header values are trimmed before the blank check;
  the echoed value uses the trimmed form. Maximum length cap is **not**
  introduced — `requirements.md` §Out of scope defers all NFRs.
- The filter does not validate UUID shape of inbound IDs — callers may
  pass any non-blank string. Internally generated IDs are always UUIDs;
  the test for the "absent header" case asserts UUID shape only when no
  header was supplied.
- Patterns mirrored:
  - `src/main/java/.../audit/api/shared/` package created in T-1 (or
    here if T-2 lands first — pure additive).
  - Test approach mirrors
    `src/test/java/.../audit/api/ingestion/IngestionIT.java`
    (Testcontainers + MockMvc).

## Step-by-step execution

Test-first per Ground rule 6.

1. **Failing test (DoD: supplied header echoed):** in `IngestionIT`,
   send a valid `POST /audit-events` with `X-Correlation-Id:
   2f3c…fixed`. Assert response status `201/200` (as ingestion does
   today) and `X-Correlation-Id` equals the supplied value. Run —
   fails (no filter, no header echoed).
2. **Implement `CorrelationIdFilter`** with the supplied-header branch
   only (read header → MDC → set response header → chain → finally
   MDC.remove). Re-run — green.
3. **Failing test (DoD: generated UUID when absent):** in
   `IngestionIT`, send the same valid POST without the header. Assert
   response carries an `X-Correlation-Id` that matches the UUID regex
   `^[0-9a-fA-F]{8}-([0-9a-fA-F]{4}-){3}[0-9a-fA-F]{12}$`. Run — fails
   (filter currently returns empty header).
4. **Implement the missing-header branch:** generate UUID, MDC.put,
   set header. Re-run — green.
5. **Failing test (DoD: header present on 400):** in `IngestionIT`,
   send an invalid POST (blank `actor`) and assert response carries
   an `X-Correlation-Id`. Run — should be green if step 4 set the
   header before `chain.doFilter()`. If not, fix the ordering. This
   step also implicitly proves the filter survives whichever error
   path is active (T-1 RFC 7807 advice if merged, Spring default
   otherwise).
6. **Failing test (DoD: MDC cleanup):** in
   `CorrelationIdFilterTest`, call the filter with a stub chain that
   asserts `MDC.get("correlationId")` is non-null during the chain.
   After the filter returns, assert `MDC.get("correlationId")` is
   `null`. Also run a second invocation that throws inside the chain
   to verify MDC is cleared in the exception path too. Run — green
   once `finally` is in place; fails if cleanup is on the happy path
   only.
7. **Static checks:** `mvn spotless:apply` then `./mvnw test`.
   ArchUnit `LayeringTest` must stay green — the filter lives in
   `api/shared/`, depends only on Servlet / Spring Web, no Domain or
   Application dependencies.
8. **LOC budget check:** `git diff --stat main...HEAD` — exclude
   generated. Budget ~220 LOC; split if approaching 400.

## Risks / open questions

- **Header-name collision with reverse proxies.** Some infra adds its
  own `X-Request-Id` / `X-Trace-Id`. We follow `design.md` §2 and
  `ARCHITECTURE.md` strictly: `X-Correlation-Id` only, no second
  spelling. If an upstream proxy injects a different header, that is a
  deployment concern, not a code concern (out of scope).
- **Filter ordering with future filters.** `@Order(Ordered.HIGHEST_PRECEDENCE)`
  guarantees this filter runs first today. If a future security filter
  needs to run earlier, it can take a higher precedence number; document
  in that task, not here.
- **MDC leak in async/streaming responses.** Not relevant in v1 — no
  async endpoints exist. Note in PR description so the next async
  feature owner knows to add MDC propagation.
