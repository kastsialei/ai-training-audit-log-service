# T-1: Add Global RFC 7807 Advice

**Refs:** US-4 ([requirements.md](../requirements.md)), [design.md](../design.md) section 2 "Error response", section 7 "Prerequisites"
**Depends on:** —
**Est. LOC:** ~300

## Goal

Install the cross-cutting error-response prerequisite the query endpoint
will later depend on. After this task, every error returned by the
service uses Spring's `ProblemDetail` with `Content-Type:
application/problem+json` and the RFC 7807 field set documented in
`design.md` §2. Existing ingestion error responses (`POST /audit-events`)
migrate to the new shape in the same commit — that is intentional and
called out in `design.md` §7. **In addition, bad-input domain
validation that today surfaces as `500` (via
`InvalidAuditEventException`) is mapped to `400` here** — per the
2026-05-11 pass-2 requirements/design update (`requirements.md` US-4,
`design.md` §7 / §10). No query-specific exceptions are added here;
that is T-10's job.

## Approach

- New package `api/shared/` for cross-cutting Spring Web infra. Mirrors
  the layering convention in `ARCHITECTURE.md` (Api layer, no domain leaks).
- One `@RestControllerAdvice` class returning Spring 6 `ProblemDetail`
  via `ResponseEntity<ProblemDetail>` so headers (`Content-Type:
  application/problem+json`) and `instance` are explicit.
- Handlers, in order of specificity:
  1. `MethodArgumentNotValidException` — bean-validation failures on
     `@Valid` request bodies (used by `IngestEventRequest`).
  2. `MethodArgumentTypeMismatchException` — query param type binding
     (e.g. enum/timestamp parse). Needed by T-10 but mapped now so the
     same shape is used end-to-end.
  3. `MissingServletRequestParameterException` — required query param
     absent.
  4. `HttpMessageNotReadableException` — malformed JSON body.
  5. `InvalidAuditEventException` (domain) — mapped to `400`. Resolves
     the existing `500` bug for bad ingestion payloads.
  6. Catch-all `Exception` — wraps unhandled exceptions in a `500`
     RFC 7807 `ProblemDetail` with a generic `detail`; **never** echoes
     the exception message or stack. Per `requirements.md` §Out of scope
     ("5xx response shape"), the specific `type` / `detail` strings for
     the 5xx path are **not** a stable API contract — only the shape
     (envelope + RFC 7807 fields + `application/problem+json`) is
     stable. Tests therefore assert the envelope and leak-guard, not the
     exact `type` URI or `detail` text.
- Stable `type` URIs come from a small constants holder (`ProblemTypes`)
  under `api/shared/`. URI prefix `https://audit-log-service/problems/`
  matches `design.md` §2. T-1 adds `validation-error`,
  `malformed-request`, `invalid-audit-event` (for the domain mapping),
  and `internal-error`. T-10 will add query-specific entries.
- `instance` is populated from the current request URI so test asserts
  are deterministic.
- This task does NOT introduce `InvalidQueryException` /
  `InvalidCursorException` (T-10) or correlation-ID work (T-2). Catch-all
  swallows everything else for now.

## Step-by-step execution

Test-first per Ground rule 6.

1. **Failing test (DoD #1, #2):** in `IngestionIT`, send a `POST
   /audit-events` body with a blank `actor`. Assert status `400`,
   `Content-Type` starts with `application/problem+json`, and JSON body
   has `type`, `title`, `status`, `detail`, `instance`. Run — fails
   because no advice exists.
2. **Implement `ProblemTypes`** holding the three `type` URI constants.
3. **Implement `GlobalExceptionHandler`** with the
   `MethodArgumentNotValidException` handler first. Construct
   `ProblemDetail.forStatusAndDetail(BAD_REQUEST, summary)`, set `type`
   from `ProblemTypes`, set `instance` from
   `request.getRequestURI()`, return `ResponseEntity.status(...)
   .contentType(MediaType.APPLICATION_PROBLEM_JSON).body(...)`. Re-run
   test — green.
3. **Failing test (new DoD bullet: `InvalidAuditEventException` →
   `400`):** in `IngestionIT`, send a `POST /audit-events` body that
   passes bean validation but trips the domain invariant (e.g., a
   `context` value that the entity rejects). Today this surfaces as
   `500`. Assert `400`, `application/problem+json`, RFC 7807 fields, and
   `type` ends with `/invalid-audit-event`. Run — fails (currently
   `500`).
4. **Implement domain-exception handler.** Add a
   `@ExceptionHandler(InvalidAuditEventException.class)` method on
   `GlobalExceptionHandler` returning a `400` `ProblemDetail` with
   `type=…/invalid-audit-event`. Log at WARN (client error, not server
   error). Re-run — green.
5. **Failing test (DoD: catch-all leak guard):** add a focused test
   that triggers the catch-all (route a generic `RuntimeException`
   through a test-only controller in `src/test/java/.../api/shared/`).
   Assert status `500`, `application/problem+json`, and that body does
   **not** contain the underlying exception message or class name.
6. **Implement catch-all `Exception` handler.** Generic `detail` like
   "An unexpected error occurred." `type` from `ProblemTypes`. Log at
   ERROR with the original exception (logback already has `correlationId`
   in pattern — value is empty until T-2). Re-run — green.
7. **Implement the three framework binding handlers** (type-mismatch,
   missing-param, message-not-readable) with the same shape. Add small
   unit-level tests where practical (Spring `WebMvcTest` slice). These
   handlers exist now so the query endpoint inherits the shape later;
   they share the `validation-error` / `malformed-request` types.
8. **Confirm success path unchanged (DoD: existing ingestion success
   tests):** re-run full `IngestionIT`; the green ingestion tests must
   still pass without touching response-body assertions on success.
9. **Static checks:** `mvn spotless:apply` then `./mvnw test`. Confirm
   ArchUnit `LayeringTest` still green — `api/shared/` lives in the API
   layer so dependencies stay inward-only. Note: the advice depends on
   the domain exception type — that is allowed because API may depend
   on Domain (inward-only).
10. **LOC budget check (DoD: <400 LOC):** `git diff --stat main...HEAD`
    — exclude generated. Split if approaching 400.

## Risks / open questions

- **`InvalidAuditEventException` 500→400 fix.** Resolved 2026-05-11 in
  `_delta.md` pass 2: bad-input domain validation maps to `400` here
  (US-4 new AC, design §7 #1, tasks T-1 Scope + DoD).
- **Where to prove the catch-all without contaminating production code.**
  Prefer a test-only controller in `src/test/java/.../api/shared/` over
  injecting a throw site into production code.
- **Domain exception import from the API layer.** `GlobalExceptionHandler`
  references `InvalidAuditEventException` from `domain/ingestion/`. This
  is API→Domain (inward), which is allowed by ArchUnit `LayeringTest`.
  If T-3 promotes `InvalidAuditEventException` to `domain/shared/` later,
  update imports at that point — out of scope for T-1.
