# T-11: Harden Pagination and Concurrent-Read Tests

**Refs:** US-2, US-3 ([requirements.md](../requirements.md)), [design.md](../design.md) section 5 "Invariants", section 11 "Test strategy"
**Depends on:** T-8, T-9, T-10
**Est. LOC:** ~330

## Goal

Prove at the HTTP/integration level that the read endpoint honors the three
read invariants most likely to break silently: canonical sort order, page
stability under newer-timestamp ingestion, and exact equivalence between a
cursor walk and a single un-paginated read of the same filter. After this
task, future changes to the reader, the cursor codec, or the controller will
fail loudly the moment they violate `design.md` §5.

This is a **test-only slice** per T-11 *Out of scope* in `tasks.md`: no new
production code unless a test surfaces a real defect, in which case T-11
stops and a follow-up task fixes the bug first.

## Approach

- All coverage lives in the API layer (`*IT.java`) and is driven through
  `MockMvc + Testcontainers + Flyway`, matching the pattern in
  `IngestionIT` (`@SpringBootTest + @AutoConfigureMockMvc + @Testcontainers`).
  No reader-level shortcuts: the invariants are caller-facing, so the tests
  are caller-facing.
- Two test classes to stay under the LOC budget and to keep file sizes
  readable:
  - `AuditEventQueryPaginationIT` — multi-page walk, concurrent insert,
    `from == to` empty window. Deterministic seeds. ~150 LOC.
  - `AuditEventQueryPropertyIT` — randomized invariant (≥100 trials).
    Self-contained reproducible seed. ~140 LOC.
- One shared fixture helper, package-private, in the same `api/query/`
  test package:
  - `QueryFixtures` — seeds rows via **direct SQL** through the autowired
    `DataSource`. Direct SQL is the only realistic way to control
    `recorded_at` deterministically: the JPA entity uses
    `@Generated(EventType.INSERT)` on both `id` and `recorded_at`, which
    means Hibernate ignores any client-supplied value and Postgres
    populates them via `DEFAULT gen_random_uuid()` / `DEFAULT now()`. The
    SQL helper inserts explicit `(id, recorded_at, actor, event_type,
    resource, outcome, context)` tuples and bypasses the `@Generated`
    machinery without touching the entity. The append-only triggers
    block `UPDATE`/`DELETE` only — `INSERT` with explicit values is fine
    and ADR-0007 is not violated.
  - Helper exposes `seed(List<SeedRow> rows)` and a tiny `SeedRow` record
    so tests stay focused on intent, not JDBC plumbing. ~50 LOC.
- Fresh DB per class via Testcontainers `@Container` static field, same as
  `IngestionIT`. No cleanup between methods is needed within a class
  because each test method seeds a disjoint actor/resource namespace.
- Pagination walk and concurrent-insert tests use **fixed timestamps**
  (`OffsetDateTime.parse("2026-05-01T10:00:00Z").plusSeconds(i)`); no
  reliance on `Thread.sleep` or wall-clock precision.
- Property test uses a `Random` seeded from a constant
  (`Random.from(0xCAFEBABEL)` equivalent) so failures reproduce. A system
  property `query.property.seed` allows reseeding on demand without
  weakening the invariant — pinned in the failure message so a developer
  can rerun the exact trial.
- "No silent failures" — invariant violations must produce a descriptive
  failure that includes the seed, the active filter set, the chosen
  `limit`, the walked ID sequence, and the expected ID sequence. Plain
  `assertEquals` on lists is enough as long as the assertion message
  contains those values.

## Step-by-step execution

Test-first per Ground rule 6. In this task the *tests are the deliverable*,
so "failing first" means each new test is shown to fail in a meaningful
way (or, where the implementation is already correct, the diagnostic-quality
of the failure is exercised manually by perturbing a constant) before it is
checked in.

Set environment first (per `AGENTS.md`):

```bash
export PATH=/opt/soft/apache-maven-3.9.9/bin:$PATH
export JAVA_HOME=/opt/soft/jdk-21.0.4+7
```

1. **Create `QueryFixtures` helper.** SQL `INSERT INTO audit_events (id,
   recorded_at, actor, event_type, resource, outcome, context) VALUES (?,
   ?, ?, ?, ?::audit_outcome, ?::jsonb)`. `SeedRow` carries `UUID`,
   `OffsetDateTime`, four strings, one `Outcome`, one JSON literal. Helper
   takes an autowired `DataSource`. No assertion in the helper itself.
2. **Failing test (DoD #1, walk):** in `AuditEventQueryPaginationIT`, add
   `walkPagesWithLimit2_returnsAllRowsExactlyOnce_lastPageHasNullCursor`.
   Seed 7 rows under a unique actor `walk_actor`, timestamps `T+0s` …
   `T+6s` (descending sort means newest = `T+6s` first). Issue first
   `GET /audit-events?actor=walk_actor&limit=2` with a wide `from`/`to`
   that brackets all seeds. Walk via successive
   `GET …&cursor=<next_cursor>` until `next_cursor` is `null`. Collect IDs
   across all pages. Assertions:
   - Exactly 4 pages (`2,2,2,1`); first three have non-null `next_cursor`;
     the fourth is `null`.
   - Concatenated IDs equal the seeded IDs sorted by `(recorded_at DESC,
     id DESC)`.
   - No ID appears twice (set size equals list size).
   Run — green if T-9 / T-10 are correct; if not, the failure message
   names which page diverged.
3. **Failing test (DoD #2, concurrent insert):** add
   `newerInsertDuringWalk_doesNotAppearInRemainingPages`. Seed 5 rows
   under actor `concurrent_actor` at `T+0s` … `T+4s`. Fetch page 1
   (`limit=2`). Insert a new row under the same actor at a **strictly
   newer** `recorded_at` (`T+100s`) via the helper. Fetch the remaining
   pages using the cursor chain. Assert the new ID is **not** present
   in any page of the in-progress walk. Also assert the original 5
   rows are returned in canonical order. Per design.md §5 invariant
   #2 (2026-05-11 pass-3): the strictly-newer wording is deliberate —
   backdated writes are not a supported ingestion mode, so the
   concurrent-insert invariant only needs to be proven against
   strictly-newer rows. Document the invariant in a code comment
   referencing `design.md` §5 #2 and add a one-line comment noting
   that the **same-`recorded_at` case** (an insert tying the page's
   last row's instant) is handled by the `id DESC` keyset tiebreaker
   exercised in T-8's `tieBreaksEqualRecordedAtByIdDesc` reader test
   and re-exercised end-to-end by the property test in step 6 — no
   dedicated test for the same-instant case at this level is needed.
   Run — green by construction of the keyset predicate; failure
   indicates a real bug, in which case **stop this task** and file a
   follow-up to fix it.
4. **Failing test (DoD #4, `from == to`):** add
   `fromEqualsTo_returnsEmptyHalfOpenWindow`. Seed any few rows under
   actor `empty_window_actor` around `T`. Call
   `GET /audit-events?actor=empty_window_actor&from=T&to=T`. Assert
   `200`, `items` is `[]`, `next_cursor` is `null`. (`from` and `to`
   together count as ≥1 substantive filter per design §2 — `actor`
   is included to make intent explicit and to keep the test
   resilient if §2 is later tightened.)
5. **Build `AuditEventQueryPropertyIT` skeleton.** Class-level
   `@Container PostgreSQLContainer` (separate container from
   `AuditEventQueryPaginationIT` for isolation; both share the same
   Docker image and migrations). Autowire `MockMvc` and `DataSource`.
   `static final long DEFAULT_SEED = 0xCAFEBABEL;`. Read override from
   `System.getProperty("query.property.seed")` if present (parse as
   `Long.parseLong`).
6. **Failing test (DoD #3, randomized invariant):** add
   `cursorWalkMatchesSingleQueryOrder_over100RandomTrials`. For each
   trial `t ∈ [0, 100)`:
   1. Derive a per-trial seed = `DEFAULT_SEED ^ t`. Use it to generate
      a small dataset (size in `[5, 25]`) under a per-trial actor
      `prop_actor_t` (so trials don't cross-contaminate without needing
      DB cleanup). Vary `event_type` over a fixed pool of 3 values,
      `resource` over a pool of 3, `outcome` over `{SUCCESS, DENIED,
      ERROR, null}`. `recorded_at` = `T + random seconds in [0, 10000]`
      to make ties possible but rare.
   2. Pick a random substantive filter: actor is always present (per
      step 6.i), plus zero or one of `event_type`, `resource`, or
      `outcome` chosen at random.
   3. Pick a random `limit` in `[1, 5]`.
   4. Issue a single `GET …&limit=200` with the same filter set;
      extract the ordered ID list — this is the **oracle**.
   5. Walk the same filter set at the chosen small `limit`; collect IDs
      across pages until `next_cursor` is `null`.
   6. Assert the walked ID list equals the oracle list (exact sequence).
      On failure, fail with a message containing: trial index, derived
      seed, filter set, chosen `limit`, walked IDs, oracle IDs, and the
      first index where they diverged.
   Run — green if reader and cursor codec agree; failure pins the exact
   trial for offline reproduction via `-Dquery.property.seed=…`.
7. **Diagnostic dry-run.** Temporarily swap canonical sort to
   `(recorded_at DESC, id ASC)` in the reader (locally only, do not
   commit) and rerun T-11. Property test must fail loudly with a useful
   message. Revert. This proves the invariant is enforced by the tests,
   not by accident. Document this manual check in the PR description.
8. **LOC budget check (DoD: <400 LOC).** `git diff --stat
   main...HEAD` — exclude generated. Target ~330 LOC across the three
   files. If approaching 400, push the randomized fixture builder into
   `QueryFixtures` rather than weakening coverage.
9. **Static checks.** `mvn spotless:apply` then `./mvnw verify`. The
   latter is part of the DoD (Testcontainers runs in the `verify`
   phase). `ArchUnit` layering must stay green — all new code lives in
   `src/test/java/.../api/query/`, depends only on Spring Test / MockMvc /
   JDBC.

## Risks / open questions

- **Clock precision / fixture determinism.** Postgres `TIMESTAMPTZ` is
  microsecond resolution; Java `OffsetDateTime` carries nanos. Mixing
  `now()` with explicit values can produce sub-microsecond ties that
  reorder unexpectedly. Mitigated by `QueryFixtures` using only explicit
  `OffsetDateTime` values with second-granularity in tests. Property
  test allows ties on purpose (to exercise the `id DESC` tie-break),
  but ties are *between explicit values*, never between explicit and
  generated values.
- **Test runtime.** 100 trials × (seed + walk + oracle query) is the
  hot path. Each trial issues O(N/limit) HTTP calls; with N ≤ 25 and
  `limit ≥ 1` that is ≤ 25 calls per trial → ~2.5k MockMvc calls. In
  MockMvc this is well under a minute on a developer laptop. If it
  becomes an issue, drop to N ≤ 15 — do not drop trial count below 100
  (DoD floor).
- **Per-class Testcontainer cost.** Two `*IT` classes mean two
  container starts. Acceptable in this codebase (only one other `*IT`
  today). If the project later adopts a shared `@Testcontainers` base
  class, both new ITs migrate to it in a follow-up.
- **If a property test exposes a real defect.** Per T-11 Out of scope:
  STOP this task, file a follow-up against T-7 / T-8 / T-9 to fix the
  bug, then come back. Do not weaken the invariant or skip the failing
  trial.
- **System-property override leakage between runs.** `query.property.seed`
  is read once per class load; CI sets neither default nor override.
  Document the override convention in the PR body so the next on-call
  knows how to reproduce a flake.
