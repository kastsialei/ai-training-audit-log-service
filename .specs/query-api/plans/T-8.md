# T-8: Implement Database Query Reader

**Refs:** US-1, US-2 ([requirements.md](../requirements.md)), [design.md](../design.md) section 3 "Read patterns covered", section 5 "Sort stability" / invariants 1 and 2, section 6 "from/to corner cases"
**Glossary:** [../glossary.md](../glossary.md) — canonical names, types, exceptions, problem types
**Depends on:** T-3, T-4, T-7
**Est. LOC:** ~360

## Goal

Land the infrastructure adapter that fulfils the `EventReader` port from T-7
by reading immutable audit events out of Postgres. After this task, the
application layer can ask for a single contiguous DB slice of the canonical
result set — exact-match filters AND-ed together, half-open time bounds,
keyset-paginated under `(recorded_at DESC, id DESC)` — and get rows back in
the order callers will see them. No HTTP wiring, no problem-type mapping, no
metrics — those belong to T-9 / T-10.

This task is the "the SQL is right" gate. If the keyset predicate is wrong,
invariant 2 (page stability) breaks for the whole feature; if the sort
expression is wrong, invariant 1 (sort stability) breaks. Both are proven
here, at the layer where they originate, *not* later through the HTTP test
surface.

## Approach

- Adapter lives in `infrastructure/persistence/query/` per
  `AGENTS.md` "Where things live" and `ARCHITECTURE.md` layering. It depends
  on the application port `EventReader` (inward dependency from
  Infrastructure → Application, which is the allowed direction).
- Class: `JpaEventReader` — `@Repository`, constructor-injected
  `EntityManager`. Naming chosen over `DatabaseEventReader` because the
  adapter is JPA-specific; if a future second adapter appears (e.g. a JDBC
  one), the name still reads cleanly.
- **Query construction strategy: JPA Criteria API.** Justification:
  - Filter set is small and fixed (six predicates + optional keyset +
    fixed sort + limit) — no need for the heavier Spring Data
    `Specification` abstraction.
  - Type-safe (no string concatenation; satisfies Notes guidance:
    "Prefer structured JPA/SQL APIs over ad hoc string concatenation").
  - Plays well with the existing JPA setup that already maps `AuditEvent`
    (the Postgres `audit_outcome` enum and `jsonb` `context` are already
    Hibernate-mapped on the entity).
  - The two awkward bits are (a) the keyset row-value comparison and (b)
    the descending sort. Criteria handles `orderBy(cb.desc(...))` cleanly.
    Row-value comparison is not first-class in Criteria, so the keyset is
    written as the explicit OR form
    `recordedAt < posRt OR (recordedAt = posRt AND id < posId)`, which
    Postgres recognises as equivalent and uses the composite index for.
- **Filter predicates.** `QueryCriteria` (from `application.query`, per
  glossary §2) carries exactly five fields:
  `String actor, String resource, String eventType, Outcome outcome,
  TimeRange timeRange`, where `TimeRange` is
  `record TimeRange(OffsetDateTime from, OffsetDateTime to)` with both
  components nullable. Build a `List<Predicate>` from non-null criteria
  fields. AND them with `cb.and(predicates.toArray(new Predicate[0]))`.
  Mapping (input field → predicate):
  - `actor` → `cb.equal(root.get("actor"), criteria.actor())`
  - `resource` → `cb.equal(root.get("resource"), criteria.resource())`
  - `eventType` → `cb.equal(root.get("eventType"), criteria.eventType())`
  - `outcome` (shared `Outcome` enum from `domain.shared`, post-T-3
    promotion) → `cb.equal(root.get("outcome"), criteria.outcome())`
  - `timeRange.from()` (`OffsetDateTime`, nullable) →
    `cb.greaterThanOrEqualTo(root.get("recordedAt"), from)` —
    inclusive (`recorded_at >= from`)
  - `timeRange.to()` (`OffsetDateTime`, nullable) →
    `cb.lessThan(root.get("recordedAt"), to)` — exclusive
    (`recorded_at < to`)
  - Null criteria fields are simply skipped. Each side of the time
    window is independent: `from=null` omits the lower bound,
    `to=null` omits the upper bound. `from == to` is permitted —
    predicate evaluates to false for every row, result is empty.
- **Keyset predicate.** T-7 passes an `Optional<KeysetPosition>
  position` where
  `record KeysetPosition(OffsetDateTime recordedAt, UUID id)`. The
  adapter consumes it via `position.ifPresent(pos -> { ... })` to
  append the keyset predicate. Stays as the explicit OR form (no
  row-value comparison):
  ```
  cb.or(
      cb.lessThan(root.get("recordedAt"), pos.recordedAt()),
      cb.and(
          cb.equal(root.get("recordedAt"), pos.recordedAt()),
          cb.lessThan(root.get("id"), pos.id())
      )
  )
  ```
  i.e. `recorded_at < posRt OR (recorded_at = posRt AND id < posId)`.
  Strict-less-than on both legs. This is the exact predicate design §5
  invariant 2 demands.
- **Sort.** `cb.orderBy(cb.desc(root.get("recordedAt")), cb.desc(root.get("id")))`.
  Two-key sort; no ties remain because `id` is unique.
- **Row limit.** `em.createQuery(cq).setMaxResults(rowLimit).getResultList()`.
  The use case is responsible for passing `limit + 1` for next-cursor
  detection (T-7 contract); this adapter just honours the number it gets.
- **No paging metadata returned.** The adapter returns `List<AuditEvent>`
  (from `domain.shared`, post-T-3 promotion) in canonical order. No
  total count, no offset, no extra wrapper — T-7's use case wraps this
  into its `QueryPage` / `next_cursor` shape.

### Port shape (per glossary §3)

T-7 declares `EventReader` in `application.query`. The signature, taken
verbatim from glossary §3, is:

```java
package net.sam.ai.engineering.audit.application.query;

import net.sam.ai.engineering.audit.domain.shared.AuditEvent;

public interface EventReader {
    List<AuditEvent> read(QueryCriteria criteria, Optional<KeysetPosition> position, int rowLimit);
}
```

Imports for this adapter: `QueryCriteria` (NOT `EventCriteria`),
`KeysetPosition`, and `QueryCriteria.timeRange()` (a `TimeRange`
record) all from `application.query`; `AuditEvent` and `Outcome` from
`domain.shared` (post-T-3 promotion). `position` is `Optional<KeysetPosition>`
(NOT a nullable reference).

## Step-by-step execution

Test-first per Ground rule 6. The order below seeds tests in roughly
DoD-bullet order so that each implementation step makes a specific test
flip green.

1. **Scaffold the IT class (no assertions yet).** Create
   `JpaEventReaderIT` with the Testcontainers base, an autowired
   `EntityManager`, and a tiny `seedEvent(...)` helper that writes
   directly via `EntityManager.persist` (NOT via the HTTP endpoint —
   tests must control `recorded_at`/`id` for tie-break and keyset
   cases). Because the entity uses
   `@Generated(EventType.INSERT)` for `id` and `recorded_at`, the
   helper will need native SQL to seed rows with **explicit** values
   for the two server-generated columns. Use
   `em.createNativeQuery("INSERT INTO audit_events (id, recorded_at, actor, event_type, resource, outcome, context) VALUES (?, ?, ?, ?, ?, CAST(? AS audit_outcome), CAST(? AS jsonb))")`.
   This is necessary specifically because the deterministic UUID
   tie-break test and the keyset test both need control over `id` and
   `recorded_at`.
2. **Failing test (DoD #1 — AND filter semantics).**
   `readsOnlyRowsMatchingAllFilters_AndSemantics`: seed 6 rows that
   each violate exactly one of `actor=u_42`, `resource=doc:9821`,
   `eventType=doc.read`, `outcome=SUCCESS`, plus one row that matches
   all four. Call reader with all four filters set, `from=null`,
   `to=null`, `position=null`, `limit=10`. Assert the result is a
   single-element list whose `id` is the matching row's id. Run —
   fails because `JpaEventReader` doesn't exist.
3. **Implement `JpaEventReader` minimal viable.** Create the class with
   constructor `JpaEventReader(EntityManager em)`. Implement `read(...)`
   with only the four equality predicates, no time bounds, no keyset,
   no sort, no limit yet. Re-run step 2 — green.
4. **Failing test (DoD #3 — `from` inclusive, `to` exclusive,
   independently optional).** Four sub-cases in one test
   `appliesHalfOpenTimeWindow`:
   - Seed three rows at `recorded_at` = `T`, `T+1ms`, `T+2ms` (all
     other filter columns identical so they all match the equality
     predicates we'll set).
   - Call with `from=T, to=T+2ms` → assert exactly the rows at
     `T` and `T+1ms` come back (proves `from` inclusive, `to`
     exclusive).
   - Call with `from=T, to=null` → assert all three rows come back
     (proves `to`-only-omitted is unbounded above).
   - Call with `from=null, to=T+1ms` → assert only the row at `T`
     comes back (proves `from`-only-omitted is unbounded below).
   - Call with `from=T+1ms, to=T+1ms` → assert empty result
     (proves `from == to` is permitted and yields empty per
     design §6).
   Run — fails (time predicates not implemented yet).
5. **Implement time-bound predicates.** Add the `>= from` and `< to`
   predicates with null guards in `JpaEventReader.read(...)`. Re-run
   step 4 — green.
6. **Failing test (DoD #2 — newest-first ordering).**
   `returnsRowsNewestFirstByRecordedAtDesc`: seed three rows with
   strictly-increasing `recorded_at` and the same filterable values.
   Call reader, assert returned order is reverse-chronological. Run
   — fails (no `orderBy` yet).
7. **Implement sort.** Add
   `cb.orderBy(cb.desc(recordedAt), cb.desc(id))`. Re-run step 6 —
   green.
8. **Failing test (DoD #2 — deterministic UUID tie-break).**
   `tieBreaksEqualRecordedAtByIdDesc`: seed two rows with **exactly
   equal** `recorded_at` and two `id` values chosen so one is
   lexicographically/numerically greater than the other (e.g.,
   `UUID.fromString("00000000-0000-0000-0000-000000000001")` and
   `...-000000000002`). Call reader; assert the row with the larger
   `id` comes first. This test would pass already if step 7 emitted
   both order keys correctly; run it now to confirm.
9. **Failing test (DoD #4 — keyset predicate yields next contiguous
   slice).** `keysetReturnsNextContiguousSlice`: seed five rows with
   strictly-decreasing `(recorded_at, id)` (so the canonical order is
   row1 newest, row5 oldest). Call reader once with `position=null`,
   `limit=2` → expect `[row1, row2]`. Take `(row2.recordedAt,
   row2.id)` as the position, call reader again with that position,
   `limit=2` → expect `[row3, row4]`. Then once more with
   `(row4.recordedAt, row4.id)` and `limit=2` → expect `[row5]`. No
   duplicates, no skips. Run — fails (no keyset predicate yet, returns
   the same first two rows every time).
10. **Implement keyset predicate.** Add the explicit OR form described
    in Approach. Re-run step 9 — green. Add one extra micro-assertion
    inside the same test: a position where `recordedAt` equals an
    existing row's `recordedAt` but `id` is between two existing ids,
    to confirm the equal-`recordedAt` second leg is exercised.
11. **Implement `limit`.** Already wired in via `setMaxResults` for
    step 9 to pass; double-check the cap is honoured for `limit=1`
    and for `limit` larger than the result set (returns the full
    set with no error). Add a tiny inline assertion in the keyset
    test if not already implicit.
12. **Failing test (DoD #5 — index usage via `EXPLAIN`).**
    `actorTimeWindowQueryUsesActorRecordedAtIdIndex`: this is the
    riskiest test in the plan — Postgres's planner will pick a seq
    scan on a near-empty table. Approach:
    - Bulk-insert 1500 rows with `actor='u_42'` and 1500 rows with
      `actor='u_other'`, all spread across a one-hour `recorded_at`
      window. Wrap in a single `INSERT INTO ... SELECT
      gen_random_uuid(), now() - (s || ' ms')::interval, ...
      FROM generate_series(1, 1500) s` per actor.
    - `ANALYZE audit_events` to ensure planner stats reflect the
      seeded data.
    - Open a raw JDBC connection from the autowired `DataSource`,
      run `SET LOCAL enable_seqscan = off` in the same transaction
      as belt-and-braces (Notes: planner may still pick seq scan on
      tiny test tables — this guarantees the test answers "*can*
      this query use the index", which is what the DoD asks).
    - Issue
      `EXPLAIN (FORMAT JSON) SELECT * FROM audit_events WHERE
      actor = ? AND recorded_at >= ? AND recorded_at < ?
      ORDER BY recorded_at DESC, id DESC LIMIT 50`
      with the same parameter shape the Criteria query produces.
      (Reading the literal SQL Criteria emits is brittle; the
      EXPLAIN-shaped statement encodes the same filter columns and
      sort.)
    - Parse the returned JSON (Jackson is already on the classpath)
      and walk the plan tree. Assert at least one node has
      `Node Type` containing `Index Scan` (or `Index Only Scan`) and
      `Index Name` equals `idx_audit_events_actor_recorded_at_id`.
    Run — should pass once T-4's indexes are in place.
13. **`mvn spotless:apply`**, then
    `mvn -pl . -am verify -Dit.test=JpaEventReaderIT` (Testcontainers
    via JUnit 5 + Flyway). Confirm all six tests above are green and
    no other ITs regress. Java env per AGENTS.md:
    `export PATH=/opt/soft/apache-maven-3.9.9/bin:$PATH` and
    `export JAVA_HOME=/opt/soft/jdk-21.0.4+7`.
14. **Static & layering.** Run the existing ArchUnit `LayeringTest`
    (under `src/test/java/.../architecture/`). The new adapter must
    only depend on Application layer (`EventReader` port) and
    Domain layer (`AuditEvent`, `Outcome`) — no API imports. Confirm
    green.
15. **LOC budget check (DoD #6 — diff under 400 LOC).**
    `git diff --stat main...HEAD -- src` (exclude generated). Budget
    is ~360 LOC; if approaching 400, follow the Notes guidance: split
    the IT class into two (filter/sort/keyset in one commit, EXPLAIN
    index-usage in a follow-up commit) before merging. See Risks
    section for the split criteria.

## Risks / open questions

- **Glossary alignment.** This plan adopts
  [../glossary.md](../glossary.md) as the source of truth for type
  names, packages, port signatures, and exception structure. Where
  earlier drafts of this plan referred to `EventCriteria` or a
  nullable `KeysetPosition`, the glossary's `QueryCriteria` and
  `Optional<KeysetPosition>` win; this plan has been updated
  accordingly.
- **EXPLAIN test flakiness on small tables.** Postgres may still
  pick a seq scan even with 1500+ rows on a fresh Testcontainer if
  cost estimates favour it. Mitigations applied in step 12:
  `ANALYZE` first, plus `SET LOCAL enable_seqscan = off` in the same
  transaction. Both are session-local — they don't affect the rest
  of the suite. If the test becomes a maintenance burden, fall back
  to asserting the index exists (T-4 already proves that) plus
  *forcing* `enable_seqscan = off` and asserting the planner picks
  the specific composite index, which still answers "the query can
  use this index".
- **Splitting the EXPLAIN test out.** Notes in tasks.md §T-8 permit
  splitting adapter tests from implementation if LOC grows too
  large. Concrete split criterion: if step 12 alone pushes the LOC
  count past ~380 (bulk-seed helper + JSON walk), move *only* the
  EXPLAIN test into a follow-up commit `T-8 (follow-up): EXPLAIN
  index-usage test for JpaEventReader`. The follow-up adds no
  production code and is reviewed as a pure test addition. All
  other tests (DoD bullets 1–4) ship together with the
  implementation in the primary commit.
- **JPA Criteria vs. row-value tuple comparison.** Postgres natively
  supports `(recorded_at, id) < (?, ?)`, which is the cleanest
  expression of the keyset predicate. JPA Criteria has no first-class
  row-value comparison, so this plan uses the explicit OR form. The
  Postgres planner treats both forms identically for index use, so
  this is purely a code-readability concern. If a future refactor
  drops Criteria for `JPQL` or native SQL, prefer the tuple form
  then.
- **Seeding rows with explicit `id` / `recorded_at`.** Production
  inserts go through Hibernate, which deliberately lets Postgres
  generate both columns (per `@Generated(EventType.INSERT)`). Tests
  here need explicit values for the tie-break and keyset cases. The
  native-SQL helper in step 1 is the right tool; do **not** soften
  the entity's generation strategy for tests.
- **Empty-result correctness for `from == to`.** This is *not* an
  error — design §6 and US-2 both make it explicit. The reader
  doesn't reject it; the use case (T-7) doesn't reject it either
  (T-5 explicitly allows `from == to`). Step 4 makes the contract
  visible.
- **What if T-7's port signature differs from this plan's
  assumption?** Adopt T-7's signatures on day 1. The Criteria body
  doesn't change.
